What You'll Learn

This module covers Microsoft Entra ID (formerly Azure Active Directory) — the backbone of identity and access management in Azure. You'll understand how tenants, users, and groups work together to control who can access your cloud resources.

  • Understand what a tenant is and how it relates to your organization
  • Create and manage user accounts (internal and guest)
  • Create and manage groups for organized access control
  • Know the difference between Security and Microsoft 365 groups
🔑

What is Microsoft Entra ID?

Microsoft Entra ID is Microsoft's cloud-based identity and access management service. Think of it as the bouncer at the door of your Azure environment — it decides who gets in and what they can do.

Every Azure subscription is associated with a tenant. A tenant is a dedicated instance of Entra ID that represents your organization. When your company signs up for Azure, Microsoft 365, or any Microsoft cloud service, they get a tenant.

Key things to understand:

  • A tenant is NOT the same as a subscription. One tenant can have multiple subscriptions.
  • Your tenant has a primary domain (like yourcompany.onmicrosoft.com) and can have custom domains.
  • The tenant is the trust boundary — users in your tenant are "internal"; everyone else is "external."
In practice, your company likely already has a tenant from Microsoft 365. Azure uses the same tenant — so your employees already have identities that work in Azure.
👤

User Accounts

There are two types of users in Entra ID:

Internal users (Members): These are people who belong to your organization. They authenticate directly with your tenant and have full access to directory features. You create them directly in Entra ID.

Guest users (Guests): These are external people — contractors, partners, vendors — who you invite to your tenant. They authenticate with their OWN identity provider (their own company's Entra ID, a Microsoft account, Google, etc.) but get limited access to your resources.

Important properties for every user: Display name, User Principal Name (UPN — their login), Job title, Department, Usage location (required for licensing).

Tip: Always set the Usage location when creating users. Without it, you can't assign licenses — and some features won't work.
👥

Groups

Groups let you manage access for multiple users at once instead of one at a time. There are two dimensions to understand:

Group Type:

  • Security groups — Used for managing access to resources (RBAC role assignments, app permissions)
  • Microsoft 365 groups — Used for collaboration (shared mailbox, SharePoint, Teams)

Membership Type:

  • Assigned — You manually add and remove members. Simple, works everywhere.
  • Dynamic User — Members are automatically added/removed based on user attributes (like department = "Engineering"). Requires Entra ID Premium P1.
  • Dynamic Device — Same concept but for devices.

Most companies start with assigned groups and move to dynamic as they grow. At 50 users, manual management is fine. At 500, you want dynamic groups based on department and job title.

Key Takeaways

  • Tenants represent organizational instances for managing cloud service access.
  • User account types include internal and guest accounts with role-specific access.
  • Groups support both static (assigned) and dynamic membership models.
  • Security and Microsoft 365 are the primary group classifications.

Exercise 1: Identity Audit

Medium

Exercise 2: User Lifecycle Management

Create a complete user onboarding and offboarding workflow.

  1. Create a new user "test-engineer1" with job title "QA Engineer", department "Engineering".
  2. Create a security group "QA Team" and add the user.
  3. Verify the user can be found in the directory.
  4. Disable the user account (simulating offboarding).
  5. Delete the user and verify they appear in Deleted users.

Hard

Exercise 3: Group-Based Access Design

Design a group structure for a company with Engineering, Sales, and Support departments.

  1. Create three security groups, one per department.
  2. Create two users per department with appropriate job titles.
  3. Add users to their respective groups.
  4. Create a cross-functional group "All Staff" containing all users.
  5. Verify membership counts for each group.
  6. Consider: What would change if you used dynamic groups instead of assigned?

Scenario

A new contractor needs access to your Azure portal for 3 months. They have their own email. How do you add them?

ACreate a new user with your domain
BInvite as external guest user
CShare your admin credentials
Why this is correctGuest users are the right approach for external people — they authenticate with their own identity provider. Creating a domain user would give them a permanent internal identity they don't need.
Common mistakesCreating a domain user (A) gives a permanent internal identity — overkill for a 3-month contractor. Sharing credentials (C) violates zero-trust principles and makes auditing impossible.
Scenario

Your company wants groups that automatically add new Marketing department hires. What do you need?

AA Microsoft 365 group with assigned membership
BA security group and a PowerShell script
CAn Entra ID Premium P1 license and a dynamic security group
Why this is correctDynamic groups auto-populate based on user attributes like department. This requires P1 licensing. It's the Azure-native, zero-maintenance solution.
Common mistakesAssigned groups (A) require manual updates every time someone joins or leaves. A PowerShell script (B) works but isn't the Azure-native solution and requires maintenance.
Scenario

You deleted a user yesterday. Your manager says they need to be reinstated. What do you do?

ARestore them from the Deleted users blade within 30 days
BCreate a new user with the same name
CContact Microsoft support to recover the account
Why this is correctDeleted users are soft-deleted and can be restored within 30 days with all properties intact. This preserves group memberships, role assignments, and audit history.
Common mistakesCreating a new user (B) would lose group memberships, role assignments, and audit history. Microsoft support (C) is unnecessary — you can self-service this in the portal.
! Troubleshooting Spot
Troubleshoot

You invited a guest user but they say they never received the invitation email. The user appears in your directory with status "Pending acceptance".

What should you check?

▶ Hint 1
Check if the email address was entered correctly in the invitation.
▶ Hint 2
Ask the guest to check their spam/junk folder — Azure invitation emails sometimes get filtered.
▶ Answer
You can resend the invitation from the user's profile. Navigate to the guest user > click "Resend invitation." Also verify their email doesn't have a typo. If the org blocks external emails, they may need to whitelist microsoft-noreply@microsoft.com.
◆ Architect's Corner
Design Decision

Your company has 500 employees across 8 departments. You need to manage access to Azure resources by department. How do you structure groups?

Option A
Create one large "All Employees" group and assign permissions per resource. Simple to set up, one group to manage.
Option B
Create a dynamic security group per department based on the department attribute. Auto-populates as users join/leave, requires P1.
Option C
Create nested static groups — department groups inside a master group. Manual membership, supports group nesting.
Why this is the best choiceDynamic groups per department scale automatically. When HR updates a user's department attribute, group membership adjusts without admin intervention. Option A gives everyone the same access (violates least privilege). Option C works but requires manual maintenance that becomes error-prone at 500+ users.
