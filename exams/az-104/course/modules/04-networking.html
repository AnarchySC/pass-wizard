What You'll Learn

Virtual networking is the foundation of Azure infrastructure. Every VM, database, and app service communicates over a network. This module teaches you how to build, segment, and secure Azure networks.

  • Create virtual networks and subnets with proper IP planning
  • Understand and configure Network Security Groups (NSGs)
  • Use Application Security Groups (ASGs) for logical grouping
  • Configure public and private DNS zones
🌐

Virtual Networks and Subnets

A Virtual Network (VNet) is your private network in Azure. It's an isolated section of the Azure cloud where you deploy resources. Nothing can communicate with your VNet unless you explicitly allow it.

Every VNet has an address space — a range of private IP addresses (like 10.0.0.0/16). Within that, you create subnets — smaller ranges that segment your network.

VNet: 10.0.0.0/16 (65,536 addresses)
├── Web Subnet:  10.0.1.0/24 (251 usable)
├── App Subnet:  10.0.2.0/24 (251 usable)
└── DB Subnet:   10.0.3.0/24 (251 usable)

Critical rules:

  • VNet address spaces MUST NOT overlap if you plan to peer them
  • Azure reserves 5 addresses per subnet (first 4 + last 1)
  • A /24 gives 251 usable addresses, not 256
🛡

Network Security Groups (NSGs)

NSGs are Azure's built-in firewall. They contain rules that allow or deny traffic based on source, destination, port, and protocol.

Key concepts:

  • Rules have a priority (100-4096). Lower number = higher priority.
  • Rules are evaluated in priority order. First match wins.
  • NSGs can be attached to subnets OR network interfaces (NICs). If both exist, traffic must pass BOTH.
  • Default rules (priority 65000+) allow VNet-to-VNet traffic and deny inbound from the internet.
PriorityNameDirectionActionPort
100AllowSSHInboundAllow22
200AllowHTTPInboundAllow80,443
4096DenyAllInboundDeny*
65000AllowVNetInboundAllow*

In this example: SSH and HTTP are allowed. Everything else is denied by rule 4096 BEFORE the default AllowVNet rule at 65000 is reached.

📦

Application Security Groups (ASGs)

ASGs let you group VMs logically (like "web servers" or "database servers") and use those groups in NSG rules instead of IP addresses.

Without ASGs: "Allow traffic from 10.0.1.4, 10.0.1.5, 10.0.1.6 to port 80" — breaks when IPs change.

With ASGs: "Allow traffic from ASG-WebServers to port 80" — works regardless of IP changes.

📑

Azure DNS

Azure DNS hosts DNS zones so you can manage DNS records in Azure.

Public DNS zones resolve names from the internet. You create records (A, CNAME, TXT, MX) and point your domain's nameservers to Azure's DNS servers.

Private DNS zones resolve names only within linked VNets. They're for internal name resolution — like db.internal.myapp.com resolving to a private IP. No one outside your VNet can see these records.

Note: Private DNS zones must be explicitly linked to each VNet that needs to use them. Creating the zone alone isn't enough.

Lab 04 — Implement Virtual Networking

Estimated time: 50 minutes

1

Create Virtual Network with Subnets via Portal

▼

Create the core services virtual network using the Azure Portal.

  1. Navigate to Virtual Networks and click + Create.
  2. Configure the basics:
    SettingValue
    Resource Groupaz104-rg4 (create new)
    NameCoreServicesVnet
    RegionEast US
  3. On the IP Addresses tab, set the address space to 10.20.0.0/16.
  4. Delete the default subnet.
  5. Add subnet SharedServicesSubnet with address range 10.20.10.0/24.
  6. Add subnet DatabaseSubnet with address range 10.20.20.0/24.
  7. Click Review + create, then Create.
  8. Once deployed, go to the resource and click Export template to save the ARM template for reuse.
Note Exporting the template captures the full configuration of your VNet including subnets, which you can modify and redeploy for similar environments.
2

Create Virtual Network Using Template

▼

Modify the exported template to deploy a second virtual network for a manufacturing environment.

  1. Open the exported template JSON file from Task 1.
  2. Replace all occurrences of CoreServicesVnet with ManufacturingVnet.
  3. Replace the address space 10.20.0.0 with 10.30.0.0 (keep /16 prefix).
  4. Update the subnets:
    Subnet NameAddress Range
    SensorSubnet110.30.20.0/24
    SensorSubnet210.30.21.0/24
  5. Navigate to Deploy a custom template in the Azure Portal.
  6. Select Build your own template in the editor, paste the modified JSON, and click Save.
  7. Select the az104-rg4 resource group and region East US, then deploy.
  8. Verify both VNets appear in the resource group after deployment.
Tip Using exported templates speeds up infrastructure replication and ensures consistency across environments.
3

Configure ASG and NSG

▼

Create an application security group and a network security group with custom rules to control traffic flow.

  1. Navigate to Application security groups and create one named asg-web in az104-rg4.
  2. Navigate to Network security groups and create one named myNSGSecure in az104-rg4.
  3. Associate myNSGSecure with SharedServicesSubnet in CoreServicesVnet.
  4. Create an inbound security rule:
    SettingValue
    NameAllowASG
    SourceApplication security group — asg-web
    Destination port ranges80,443
    ProtocolTCP
    ActionAllow
    Priority100
  5. Create an outbound security rule:
    SettingValue
    NameDenyInternetOutbound
    DestinationInternet
    ServiceCustom (all ports)
    ActionDeny
    Priority4096
  6. Review the effective security rules to confirm the new rules appear alongside the defaults.
Warning The DenyInternetOutbound rule at priority 4096 blocks all outbound Internet traffic. This can prevent VMs from reaching external update servers or Azure services. Use with caution and add explicit allow rules for required traffic.
4

Configure DNS Zones

▼

Set up public and private DNS zones and verify name resolution.

  1. Navigate to DNS zones and create a public zone:
    SettingValue
    Namecontoso.com
    Resource Groupaz104-rg4
  2. Add an A record: Name www, IP address 10.1.1.4.
  3. Verify the record with nslookup:
    CLI nslookup www.contoso.com <Azure-NS-server>
  4. Navigate to Private DNS zones and create a zone named private.contoso.com.
  5. Add a Virtual network link to ManufacturingVnet (enable auto-registration if desired).
  6. Add an A record: Name sensorvm, IP address 10.1.1.4.
  7. Verify the private record resolves from a VM within the linked VNet.
Note Private DNS zones resolve only for resources within linked virtual networks. External queries will not return results for private records.

Key Takeaways

  • Virtual networks represent your cloud network infrastructure and isolate resources within Azure.
  • Avoiding overlapping IP address ranges across VNets reduces troubleshooting complexity when peering or connecting networks.
  • NSGs contain customizable allow/deny traffic rules that filter traffic at the subnet or NIC level.
  • Application Security Groups let you protect servers with common functions without managing individual IP addresses.
  • Azure DNS provides name resolution hosting for both public and private domains within your infrastructure.

Exercises

Easy

Exercise 1: IP Address Planning

  1. Given a company with 3 departments (HR: 50 hosts, Dev: 200 hosts, Ops: 30 hosts), design an address space.
  2. Choose a /16 address space and create appropriately-sized subnets for each department.
  3. Document the subnet ranges and usable host counts.
  4. Verify no subnets overlap.
Show Hint
A /24 gives 251 usable addresses, /25 gives 123, /26 gives 59, /27 gives 27. Remember that Azure reserves 5 addresses per subnet, so the usable count is 2^(32-prefix) - 5.
Medium

Exercise 2: NSG Rule Analysis

  1. Create an NSG and review the default rules (priorities 65000–65500).
  2. Create a rule allowing SSH (port 22) from your IP only, priority 100.
  3. Create a rule denying all other SSH traffic, priority 200.
  4. Create a rule allowing HTTP/HTTPS from anywhere, priority 300.
  5. Walk through: What happens when traffic arrives on port 22 from a different IP? Why?
Show Hint
NSG rules are processed in priority order (lowest number = highest priority). First match wins. Traffic on port 22 from a non-matching IP skips rule 100, hits the deny at 200, and is dropped before any later rules are evaluated.
Hard

Exercise 3: DNS Resolution Challenge

  1. Create a public DNS zone for a domain you control (or use a test domain).
  2. Add an A record, a CNAME record, and a TXT record.
  3. Create a private DNS zone and link it to a VNet.
  4. Add records to the private zone.
  5. Test resolution: use nslookup for the public zone with Azure's nameservers.
  6. Explain: Why can't you resolve private DNS records from outside the linked VNet?
Show Hint
Private DNS zones only resolve for resources within linked virtual networks. They use Azure's internal DNS at 168.63.129.16, which is only accessible from within Azure VNets.

Scenario

You created two VNets: 10.20.0.0/16 and 10.20.0.0/16 (same range). Can you peer them?

AYes — Azure will handle routing between identical ranges
BNo — peered VNets cannot have overlapping address spaces
CYes — but only if they're in different regions
Why this is correctAzure VNet peering requires non-overlapping address spaces. This is a hard requirement regardless of region. Plan your IP ranges carefully before creating VNets.
Common mistakesAzure cannot route between identical address spaces (A) — it wouldn't know which VNet to send traffic to. Region doesn't matter (C) — overlapping ranges are blocked in both local and global peering.
Scenario

You have an NSG with these inbound rules: Allow SSH (port 22) at priority 100, Deny All at priority 200. Web traffic on port 80 is being blocked. Why?

AThe NSG is broken
BSSH rules automatically block other ports
CThe "Deny All" rule at priority 200 is blocking port 80 before the default AllowVNetInBound rule (65000) can allow it
Why this is correctNSG rules process in priority order (lowest number first). Your explicit Deny All at 200 fires before the default rules at 65000+. You need to add an Allow HTTP rule at a priority lower than 200 (like 150).
Common mistakesNSGs don't break (A) — they work exactly as configured. SSH rules don't affect other ports (B) — each rule is evaluated independently based on port and priority.
Scenario

You created a private DNS zone but VMs in your VNet can't resolve records from it. What's missing?

AThe private DNS zone isn't linked to the VNet
BYou need a public IP on each VM
CPrivate DNS zones only work with premium VNets
Why this is correctPrivate DNS zones must be explicitly linked to each VNet that needs to resolve records. Without the link, the VNet's DNS resolver doesn't know about your zone.
Common mistakesPublic IPs (B) are for internet-facing resources — private DNS is internal only. There's no such thing as "premium VNets" (C) — all VNets support private DNS zones.
! Troubleshooting Spot
Troubleshoot

You associated an NSG with a subnet. You added a rule to allow HTTP on port 80 with priority 100 and action Allow. But web traffic still can't reach your VMs.

What's going wrong?

▶ Hint 1
Check if there's also an NSG on the VM's NIC — traffic must pass BOTH the subnet NSG and the NIC NSG.
▶ Hint 2
Check the Source setting on your rule — if it's set to a specific IP or service tag, traffic from other sources will be denied.
▶ Answer
Most likely there's a second NSG on the network interface. NSGs are evaluated independently at both subnet and NIC levels. Traffic must be allowed by BOTH. Check the NIC-level NSG and either add an allow rule there too, or remove the NIC NSG to rely solely on the subnet NSG.
◆ Architect's Corner
Design Decision

You're designing the network for a company with web servers (public-facing), application servers (internal), and database servers (highly restricted). How do you structure your VNet?

Option A
One VNet, one subnet, use NSG rules to separate traffic. Simple layout, all security via NSG rules.
Option B
One VNet, three subnets (web, app, db), each with its own NSG. Network segmentation with defense in depth.
Option C
Three separate VNets, one per tier. Maximum isolation, requires peering for communication.
Why this is the best choiceThree subnets in one VNet gives you segmentation without peering complexity. The web subnet NSG allows inbound HTTP/HTTPS from the internet. The app subnet only allows traffic from the web subnet. The db subnet only allows traffic from the app subnet on the database port. This is classic defense-in-depth. Option A works but one flat subnet makes it easier to misconfigure. Option C adds unnecessary peering overhead since these tiers need to communicate constantly.
