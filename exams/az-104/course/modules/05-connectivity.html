Administer Intersite Connectivity

Configure virtual network peering, test connectivity with Network Watcher, and implement custom routing with route tables.

What You'll Learn

When your Azure resources live in separate virtual networks, they can't talk to each other by default. This module teaches you how to connect VNets using peering, verify connectivity with Network Watcher, and control traffic flow with custom routes.

  • Understand why VNets are isolated by default
  • Configure VNet peering for cross-network communication
  • Use Network Watcher to diagnose connectivity issues
  • Create user-defined routes to control traffic paths
🔗

VNet Isolation and Peering

By default, VNets are completely isolated — even if they're in the same subscription, region, and resource group. A VM in VNet-A cannot reach a VM in VNet-B without explicit configuration.

VNet Peering creates a direct connection between two VNets using Microsoft's backbone network (not the public internet). Traffic between peered VNets is private, low-latency, and high-bandwidth.

Two types:

  • Regional peering — VNets in the same Azure region
  • Global peering — VNets in different regions (slightly higher latency)

Critical rules:

  • Peering is non-transitive. If A peers with B, and B peers with C, A CANNOT reach C through B.
  • Address spaces must NOT overlap
  • Peering must be created from BOTH sides — it's a two-step process
  • Both sides must show "Connected" status before traffic flows
 VNet-A <──peer──> VNet-B <──peer──> VNet-C
   ↑                                    ↑
   ←── Cannot reach each other ────────→
        (peering is non-transitive)
👁

Network Watcher

Network Watcher is Azure's built-in network diagnostic toolkit. The key tool is Connection Troubleshoot — it tests whether one VM can reach another on a specific port and shows you exactly where traffic is blocked.

What it checks:

  • Network path between source and destination
  • NSG rules at every hop
  • Route tables and system routes
  • Whether the destination port is listening
Real-World: Before troubleshooting complex network issues manually, always run Connection Troubleshoot first. It often pinpoints the exact NSG rule or missing route that's blocking traffic in seconds.
🛣

User-Defined Routes (UDRs)

Azure automatically creates system routes so VNets, subnets, and the internet can communicate. You can override these with User-Defined Routes in a route table.

Common use case: Force all outbound traffic through a firewall/NVA (Network Virtual Appliance) for inspection.

How it works:

  1. Create a Route Table
  2. Add routes (destination prefix → next hop type + address)
  3. Associate the route table with a subnet

Next hop types: Virtual network gateway, Virtual network, Internet, Virtual appliance (specific IP), None (drop traffic)

Important: A route table does NOTHING until you associate it with a subnet. Creating it and adding routes isn't enough.

Network Watcher Connection Testing

▼

Use Network Watcher to verify that the two VMs cannot communicate before peering is configured.

  1. Navigate to Network Watcher in the Azure Portal.
  2. Select Connection Troubleshoot from the left menu.
  3. Configure the test:
    SettingValue
    SourceCoreServicesVM
    DestinationManufacturingVM
    ProtocolTCP
    Destination port3389
  4. Click Check and wait for the result.
  5. Verify the result shows Unreachable status.
Expected Result The connection should fail because the two virtual networks are isolated. Without explicit peering or a gateway, traffic between separate VNets is not routed.
4

Configure Virtual Network Peering

▼

Create bidirectional peering between CoreServicesVnet and ManufacturingVnet to enable private connectivity.

  1. Navigate to CoreServicesVnet and select Peerings from the left menu.
  2. Click + Add to create a new peering.
  3. Configure the peering (this VNet side):
    SettingValue
    Peering link name (this VNet)CoreServices-to-Manufacturing
    Allow access to remote VNetEnabled
    Allow forwarded trafficEnabled
  4. Configure the remote VNet side:
    SettingValue
    Peering link name (remote VNet)Manufacturing-to-CoreServices
    Virtual networkManufacturingVnet
    Allow access to remote VNetEnabled
    Allow forwarded trafficEnabled
  5. Click Add to create the bidirectional peering.
  6. Wait for both peering links to show Connected status.
Note Azure creates peering in both directions simultaneously when you configure it from the portal. Both sides must show "Connected" status for traffic to flow. Peering uses the Microsoft backbone network — no public Internet, no gateways, no encryption needed.
5

Retest Connectivity with PowerShell

▼

Verify that peering has enabled connectivity by testing from the ManufacturingVM.

  1. Navigate to CoreServicesVM and note its private IP address from the Overview blade.
  2. Connect to ManufacturingVM via RDP or Bastion.
  3. Open PowerShell on ManufacturingVM and run:
    PowerShell Test-NetConnection <CoreServicesVM-Private-IP> -port 3389
  4. Verify the result shows TcpTestSucceeded: True.
Tip If the test still fails, verify that both peering links show "Connected" status and that no NSG rules are blocking port 3389 on the destination VM.
6

Create a Custom Route

▼

Add a perimeter subnet and configure a user-defined route to direct traffic through a virtual appliance.

  1. Navigate to CoreServicesVnet and select Subnets.
  2. Add a new subnet:
    SettingValue
    Subnet nameperimeter
    Address range10.0.1.0/24
  3. Navigate to Route tables and click + Create:
    SettingValue
    Resource Groupaz104-rg5
    Namert-CoreServices
    Region(same as VNet)
  4. Open the route table and select Routes, then click + Add:
    SettingValue
    Route namePerimetertoCore
    Destination typeIP Addresses
    Destination10.0.0.0/16
    Next hop typeVirtual appliance
    Next hop address10.0.1.7
  5. Select Subnets in the route table and click + Associate.
  6. Associate the route table with the perimeter subnet in CoreServicesVnet.
Warning The virtual appliance IP (10.0.1.7) must exist and have IP forwarding enabled for traffic to actually flow through it. In this lab the route demonstrates the concept — without a real appliance, traffic routed to that IP will be dropped.

Key Takeaways

  • Virtual networks are isolated by default — without explicit peering or a gateway, VMs in different VNets cannot communicate.
  • VNet peering uses Microsoft backbone infrastructure — low latency, high bandwidth, no public Internet exposure.
  • System routes are created automatically for each subnet; user-defined routes (UDRs) override them when more specific.
  • Network Watcher provides diagnostic tools like Connection Troubleshoot to test and verify connectivity between resources.

VNet Peering Configuration

▼

Walk through configuring bidirectional VNet peering in the Azure Portal:

  2. Show the peering creation form — both the local and remote VNet settings are configured at the same time.
  3. Explain the "Allow access" and "Allow forwarded traffic" toggles and when each is needed.

Network Routing and Endpoints

▼

Demonstrate how Azure handles network routing and service endpoints:

  1. Navigate to a NIC and open Effective routes to show the system-generated route table.
  2. Identify the default routes: VNet address space, 0.0.0.0/0 (Internet), and RFC 1918 blocks (None).
  3. Show how peering adds routes automatically — the peered VNet address space appears as "VNet peering" next hop.
  4. Demonstrate Service Endpoints: enable a service endpoint for Azure Storage on a subnet and show the resulting route entry.
  5. Explain the difference between service endpoints (route-based, within Azure) and private endpoints (private IP in your VNet).
Tip Effective routes on a NIC show the combined result of system routes, UDRs, and BGP routes. This is the best tool for debugging routing issues.

Route Tables and Subnet Association

▼

Demonstrate creating and associating route tables with subnets:

  1. Create a new route table and explain the "Propagate gateway routes" setting.
  2. Add a custom route with a virtual appliance next hop type.
  3. Associate the route table with a subnet and show how it appears in the subnet configuration.
  4. Navigate to a VM NIC in the associated subnet and check Effective routes to see the UDR override the system route.

Exercises

Easy

Exercise 1: Peering Status Check

  1. Navigate to any virtual network and open the Peerings blade.
  2. If peerings exist, verify both sides show Connected status.
  3. Check if Allow forwarded traffic is enabled on both sides.
  4. Document the address spaces of both peered networks.

Medium

Exercise 2: Connectivity Diagnosis

  1. Deploy two VMs in different virtual networks (no peering configured).
  2. Use Network Watcher > Connection Troubleshoot to test TCP 3389 between them.
  3. Document the failure and the reason given.
  4. Create bidirectional peering between the VNets.
  5. Re-test connectivity and document the successful result.
Show Hint
Network Watcher's Connection Troubleshoot shows each hop in the path. Look at where the traffic is being dropped. Before peering, the traffic has no route to the destination VNet. After peering, the Microsoft backbone provides the path.
Hard

Exercise 3: Custom Routing Design

Design a hub-and-spoke network with custom routing:

  1. Create a hub VNet (10.0.0.0/16) with a "firewall" subnet (10.0.1.0/24).
  2. Create two spoke VNets (10.1.0.0/16 and 10.2.0.0/16).
  3. Peer both spokes to the hub (but not to each other).
  4. Create a route table that forces spoke-to-spoke traffic through the hub's firewall subnet.
  5. Associate the route table with both spoke subnets.
  6. Explain: Why can't Spoke1 talk to Spoke2 without the custom route?
Show Hint
VNet peering is non-transitive. Spoke1 peers with Hub and Spoke2 peers with Hub, but Spoke1 and Spoke2 can't see each other automatically. You need user-defined routes that send spoke-to-spoke traffic to a virtual appliance (firewall) in the hub, which then forwards it to the other spoke.

Scenario

VNet-A is peered with VNet-B, and VNet-B is peered with VNet-C. Can VNet-A communicate with VNet-C?

AYes — peering is transitive so traffic flows through VNet-B
BNo — VNet peering is non-transitive; A and C need their own peering or a hub route
CYes — but only if all three are in the same region
Why this is correctVNet peering is non-transitive. Each pair needs explicit peering or you need UDRs through a hub/NVA. This is fundamental to Azure networking.
Common mistakesMany assume peering works like a physical switch where all connected networks can see each other. In Azure, peering is a direct link between exactly two VNets — no transit.
Scenario

You set up peering between two VNets. One side shows 'Connected' but the other shows 'Initiated'. Can traffic flow?

AYes — one-sided peering is enough
BYes — but only in one direction
CNo — both sides must show 'Connected' for traffic to flow
Why this is correctPeering requires both sides to be configured and showing "Connected". "Initiated" means only one side has been set up. Traffic won't flow until the reciprocal peering is created.
Common mistakesPeering is not like a one-way route — it's a bilateral agreement. "Initiated" is a transitional state that means the other side hasn't accepted yet.
Scenario

You created a user-defined route that sends 10.0.0.0/16 traffic to a virtual appliance at 10.0.1.7. But traffic to 10.0.0.0/16 is still taking the default system route. Why?

AThe route table isn't associated with the source subnet
BUser-defined routes have lower priority than system routes
CThe virtual appliance must be a VM with a public IP
Why this is correctRoute tables only apply when associated with a subnet. Creating the table and adding routes isn't enough — you must associate it with the subnet whose traffic you want to redirect. UDRs DO override system routes when properly associated.
Common mistakesUDRs actually have HIGHER priority than system routes (B is wrong). The virtual appliance doesn't need a public IP (C is wrong) — it just needs IP forwarding enabled.
! Troubleshooting Spot
Troubleshoot

You peered two VNets and both show 'Connected'. But VMs still can't communicate on TCP port 3389 (RDP).

What's wrong?

▶ Hint 1
Peering enables network-level reachability, but NSGs still apply. Check if either VM's NSG blocks port 3389.
▶ Hint 2
Check if the VMs have public IPs and you're testing against the public IP. Peering works on private IPs only.
▶ Answer
Most likely an NSG is blocking port 3389. Peering makes the networks reachable but doesn't override security rules. Check inbound NSG rules on the destination VM's subnet and NIC. Also ensure you're using the private IP address — peering routes private traffic only, not public IP traffic.
◆ Architect's Corner
Design Decision

You have 5 workload VNets that all need to talk to each other and to a shared services VNet (DNS, monitoring). How do you connect them?

Full mesh peering — peer every VNet to every other VNet (15 peerings)
Direct connections, lowest latency, most peerings to manage
Hub-and-spoke — peer all workloads to shared services hub, use UDRs for spoke-to-spoke
Centralized, scalable, requires NVA/firewall in hub
One giant VNet with subnets instead of separate VNets
No peering needed, simpler networking
Why this is the best choiceHub-and-spoke is the Azure-recommended pattern for this scenario. The hub hosts shared services and an NVA/firewall that routes spoke-to-spoke traffic. You manage only 5 peerings instead of 15, and adding new spokes is simple. Full mesh (A) becomes unmanageable as VNets grow. One giant VNet (C) loses the isolation and security benefits that separate VNets provide.
