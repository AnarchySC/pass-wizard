Manage subscriptions, RBAC, Azure Policy, resource tagging, and resource locks for organizational governance.

What You'll Learn

Governance is about keeping your Azure environment under control — who can do what, where resources can live, and how you organize everything. This module covers management groups, RBAC, Azure Policy, tags, and resource locks.

  • Understand the Azure resource hierarchy (management groups → subscriptions → resource groups → resources)
  • Assign and create RBAC roles for least-privilege access
  • Use Azure Policy to enforce organizational standards
  • Apply tags and resource locks for governance
🏢

The Azure Resource Hierarchy

Azure organizes resources in a hierarchy. Understanding this is critical because permissions and policies flow downward:

Management Groups
  └── Subscriptions
        └── Resource Groups
              └── Resources (VMs, Storage, etc.)
  • Management Groups — Containers for subscriptions. Let you apply policy and RBAC across multiple subscriptions at once.
  • Subscriptions — Billing boundaries. Each subscription gets a separate bill.
  • Resource Groups — Logical containers for related resources. A resource group is NOT a security boundary by default — it's for organization.
  • Resources — The actual things (VMs, databases, storage accounts).

Key rule: Permissions assigned at a higher level are inherited by everything below. Give someone Reader on a subscription? They can read every resource in every resource group in that subscription.

🔒

Role-Based Access Control (RBAC)

RBAC answers the question: "Who can do what, and where?"

Three built-in roles cover most needs:

  • Owner — Full access + can assign roles to others
  • Contributor — Full access EXCEPT assigning roles
  • Reader — View only, can't change anything

Beyond these, there are 100+ specialized roles like Virtual Machine Contributor (manage VMs only) and Storage Blob Data Reader (read blobs only).

Best practice: Always assign roles to groups, not individual users. When someone joins or leaves, you just update group membership — no need to touch role assignments.

Important: Owner role does NOT bypass Azure Policy. Even an Owner can't create a resource that violates an active policy. This is by design — Policy is pre-deployment enforcement.
📜

Azure Policy

Azure Policy enforces rules on your resources. It answers: "What is allowed in this environment?"

How it works:

  1. A Policy Definition describes what to check and what to do (written in JSON)
  2. You create a Policy Assignment that applies the definition to a scope (management group, subscription, or resource group)
  3. Azure evaluates every resource operation against active policies
  4. Non-compliant operations are denied, flagged, or auto-remediated

Common built-in policies:

  • "Allowed locations" — Restrict where resources can be deployed
  • "Require a tag and its value" — Force tagging on new resources
  • "Allowed virtual machine SKUs" — Restrict VM sizes
Policy is your first line of defense. Before a resource is even created, Policy checks if it's allowed. RBAC controls WHO can act. Policy controls WHAT actions are allowed. Locks prevent accidental destruction AFTER resources exist.
🏷

Tags and Resource Locks

Tags are key-value pairs you attach to resources for organization. They're metadata — they don't affect how resources work, but they're essential for cost management, automation, and compliance.

Common tags: Environment: Production, CostCenter: 12345, Owner: jane@company.com

Resource Locks prevent accidental changes:

  • Delete lock — Resources can be modified but not deleted
  • Read-only lock — Resources can't be modified OR deleted

Locks are inherited. A lock on a resource group applies to all resources inside it. Even Owners can't bypass a lock — you must remove the lock first, then delete.

Lab 02a - Manage Subscriptions and RBAC

Estimated time: 20 minutes

1

Implement Management Groups

▼
  1. Sign into the Azure Portal.
  2. Search for and select Management groups.
  3. Click + Create to add a new management group.
  4. Configure the management group:
    SettingValue
    Management group IDaz104-mg1
    Display nameaz104-mg1
  5. Click Submit to create the management group.
Note Management groups provide a level of scope above subscriptions, allowing you to organize subscriptions hierarchically for unified policy and access management.
2

Review and Assign a Built-in Azure Role

▼
  1. Select the az104-mg1 management group.
  2. Navigate to Access control (IAM) and click the Roles tab to review built-in roles.
  3. Click + Add > Add role assignment.
  4. Search for and select the Virtual Machine Contributor role.
  5. On the Members tab, assign access to the helpdesk group.
  6. Review and assign the role.
Best Practice Always assign roles to groups rather than individual users. This simplifies access management and reduces administrative overhead as team membership changes.
3

Create a Custom RBAC Role

▼
  1. In the az104-mg1 management group, go to Access control (IAM).
  2. Click + Add > Add custom role.
  3. Set the custom role name to Custom Support Request.
  4. Under Baseline permissions, select Clone a role and choose Support Request Contributor.
  5. On the Permissions tab, click Exclude permissions.
  6. Exclude the permission: Other: Registers Support Resource Provider.
  7. On the Review + create tab, review the JSON definition. Note the Actions, NotActions, and AssignableScopes arrays.
JSON { "Name": "Custom Support Request", "Actions": [ "Microsoft.Support/*" ], "NotActions": [ "Microsoft.Support/register/action" ], "AssignableScopes": [ "/providers/Microsoft.Management/managementGroups/az104-mg1" ] }
Note Custom roles allow you to define least-privilege access by combining Actions and NotActions to precisely control what users can and cannot do.
4

Monitor Role Assignments with Activity Log

▼
  1. Navigate back to the az104-mg1 management group.
  2. Open the Activity log blade.
  3. Review the log entries to verify that role assignment and custom role creation operations were recorded.
Tip The Activity log captures all management-plane operations, making it essential for auditing and troubleshooting governance changes.

Lab 02b - Manage Governance via Azure Policy

Estimated time: 30 minutes

1

Assign Tags via Azure Portal

▼
  1. In the Azure Portal, search for and select Resource groups.
  2. Click + Create and create a new resource group:
    SettingValue
    Resource group nameaz104-rg2
    RegionEast US
  3. After creation, open the resource group and navigate to Tags.
  4. Add a tag:
    NameValue
    Cost Center000
  5. Click Apply to save the tag.
2

Enforce Tagging via Azure Policy

▼
  1. Search for and select Policy.
  2. Under Authoring, click Definitions.
  3. Search for the built-in policy: Require a tag and its value on resources.
  4. Click Assign and configure the assignment:
    SettingValue
    Scopeaz104-rg2
    Tag NameCost Center
    Tag Value000
  5. Click Review + create, then Create.
  6. To verify, try creating a storage account in az104-rg2 without the required tag. The deployment should fail with a policy violation.
Warning Policy enforcement can take up to 30 minutes to become active after assignment. If the test deployment doesn't fail immediately, wait and retry.
3

Apply Tagging via Azure Policy

▼
  1. Navigate to Policy > Assignments.
  2. Delete the previous "Require a tag" policy assignment.
  3. Go back to Definitions and search for Inherit a tag from the resource group if missing.
  4. Assign this policy to the az104-rg2 scope with Tag Name: Cost Center.
  5. On the Remediation tab, check Create a remediation task and select the policy to remediate.
  6. Click Review + create, then Create.
  7. Create a new storage account in az104-rg2 (without manually adding tags).
  8. After deployment, verify that the Cost Center: 000 tag was automatically applied to the storage account.
Note The "Inherit" policy uses a modify effect, which automatically adds or updates tags on resources at deployment time. The remediation task handles existing resources that were created before the policy was assigned.
4

Configure and Test Resource Locks

▼
  1. Navigate to the az104-rg2 resource group.
  2. In the Settings section, click Locks.
  3. Click + Add and configure:
    SettingValue
    Lock namerg-lock
    Lock typeDelete
  4. Click OK to apply the lock.
  5. Attempt to delete the resource group. The operation should be denied with a lock conflict error.
Important Resource locks apply to all users regardless of their RBAC permissions. Even an Owner must first remove the lock before deleting a locked resource.

Key Takeaways

  • Management groups organize subscriptions hierarchically for unified governance.
  • Built-in roles provide pre-configured permissions for common scenarios.
  • Custom roles enable least-privilege access tailored to your organization.
  • Azure tags enable logical resource labeling for cost tracking and organization.
  • Azure Policy enforces rules at deployment time (pre-deployment security).
  • RBAC controls who can perform actions (post-deployment security).
  • Resource locks protect critical resources against accidental deletion or modification.

Exercise 3: Governance Strategy Design

Design a governance strategy for a company with Dev, Staging, and Production subscriptions.

  1. Create a management group hierarchy: Root > Production, Root > Non-Production.
  2. Assign "Allowed locations" policy at the Root level for East US and West US only.
  3. Create a custom role that allows VM management but prevents deletion.
  4. Apply resource locks (Delete) to the Production resource groups.
  5. Implement a tagging policy requiring "Environment" and "CostCenter" tags.
  6. Document your strategy and explain why each control exists.
Show Hint
Policies assigned at the management group level are inherited by all child subscriptions. Place shared restrictions (like allowed locations) at the Root management group, and environment-specific policies at the Production or Non-Production level. Scenario

A developer creates a VM in West Europe, but your company policy requires all resources in East US and West US only. The VM deployed successfully. What went wrong?

AAzure Policy can't restrict locations
BThe "Allowed locations" policy wasn't assigned or was assigned at the wrong scope
CThe developer has Owner role which bypasses policy
Why this is correctAzure Policy applies to everyone regardless of role — even Owners. If the VM was created, the policy either wasn't assigned, wasn't assigned at the right scope (subscription/resource group), or enforcement was set to "Disabled".
Common mistakesAzure Policy absolutely can restrict locations (A) — that's one of its most common uses. Owner role (C) does NOT bypass policy. RBAC and Policy are independent systems.
Scenario

You need to let the help desk team create support tickets and manage VMs, but nothing else. What's the best approach?

AGive them the Contributor role at the subscription level
BGive them the Owner role on a specific resource group
CCreate a custom role combining VM Contributor and Support Request Contributor permissions
Why this is correctCustom roles let you follow least-privilege. A custom role combining just the needed permissions gives them exactly what they need and nothing more.
Common mistakesContributor (A) grants too much access — can manage everything except access. Owner (B) adds the ability to grant access to others. Both violate least-privilege.
Scenario

You applied a "Delete" resource lock to a resource group. A team member tries to delete a storage account inside it and gets denied. But they CAN still modify files in that storage account. Is this correct behavior?

AYes — Delete locks prevent deletion but allow modifications
BNo — Delete locks should prevent all changes
CNo — The lock shouldn't affect resources inside the group
Why this is correctDelete locks only prevent deletion, not modification. If you need to prevent ALL changes, use a Read-only lock instead.
Common mistakesConfusing Delete locks with Read-only locks (B). Locks ARE inherited by child resources (C) — a lock on a resource group applies to all resources within it.
! Troubleshooting Spot
Troubleshoot

You assigned the "Require a tag" policy to a resource group, but users can still create resources without tags. The policy shows as "Assigned" in the portal.

What's going wrong?

▶ Hint 1
Check the policy assignment's "Enforcement" setting — it might be set to "Disabled".
▶ Hint 2
New policy assignments can take up to 30 minutes to take effect. Wait and retry.
▶ Answer
Most likely the enforcement is disabled (common when testing). Edit the assignment and set enforcement to "Enabled". Also note: policies only apply to NEW operations — existing non-compliant resources stay as-is until a remediation task is created.
◆ Architect's Corner
Design Decision

You're setting up governance for a company with dev, staging, and production environments across 3 subscriptions. Where should you assign your "Allowed locations" policy?

Option A
On each subscription individually. Direct assignment, 3 policies to manage.
Option B
On each resource group where resources are deployed. Granular control, many policies to manage.
Option C
On a management group containing all 3 subscriptions. Single policy inherited by all subscriptions.
Why this is the best choiceManagement group assignment gives you one policy to maintain that covers all current and future subscriptions. Individual subscription assignments (A) work but multiply your management overhead. Resource group level (B) is even more granular but becomes unmanageable quickly. The whole point of management groups is exactly this — centralized governance that inherits downward.
