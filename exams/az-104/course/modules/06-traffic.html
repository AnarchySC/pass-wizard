Administer Network Traffic Management

Configure Azure Load Balancer and Application Gateway to distribute and manage network traffic across backend pools.

What You'll Learn

When your app runs on multiple VMs, you need to distribute traffic between them. This module covers Azure's two main traffic distribution services: Load Balancer (Layer 4) and Application Gateway (Layer 7).

  • Understand the difference between L4 and L7 load balancing
  • Configure Azure Load Balancer with frontend, backend pool, and health probes
  • Configure Application Gateway with path-based routing
  • Know when to use each service
📑

Layer 4 vs Layer 7

Network traffic operates in layers. The two that matter here:

Layer 4 (Transport) — TCP and UDP. The load balancer sees IP addresses and port numbers, but NOT the content of the request. It can't tell the difference between a request for /images/cat.jpg and /api/users.

Layer 7 (Application) — HTTP/HTTPS. The load balancer can inspect URLs, headers, cookies, and route based on the actual content of the request.

FeatureLoad Balancer (L4)Application Gateway (L7)
OSI Layer4 (Transport)7 (Application)
ProtocolsTCP, UDPHTTP, HTTPS, WebSocket
RoutingIP + Port onlyURL path, host header, cookies
SSL terminationNoYes
WAFNoOptional (WAF tier)
PerformanceUltra-fast (no inspection)Slightly slower (inspects content)
⚖

Azure Load Balancer

Azure Load Balancer distributes incoming traffic across a pool of backend VMs. It has four components:

  1. Frontend IP — The public or private IP that receives traffic
  2. Backend Pool — The VMs that handle the traffic
  3. Health Probe — Checks if each VM is alive (TCP or HTTP)
  4. Load Balancing Rule — Maps a frontend port to a backend port

Important: Standard SKU Load Balancers block all traffic by default. You MUST have NSG rules that explicitly allow traffic, unlike Basic SKU which allows everything.

Real-World: Use Standard SKU for production. Basic is being retired and doesn't support Availability Zones, multiple frontend IPs, or SLA guarantees.
🔧

Azure Application Gateway

Application Gateway is a Layer 7 load balancer designed for web applications. Its killer features:

Path-based routing: Send /api/* requests to API servers and /images/* to image servers — from the same frontend IP.

SSL termination: HTTPS traffic is decrypted at the gateway, so backend servers only handle HTTP (simpler, faster).

WAF (Web Application Firewall): The WAF tier adds protection against SQL injection, XSS, and other OWASP top-10 attacks.

Requirements:

  • Needs its own dedicated subnet (/27 or larger)
  • Supports autoscaling (Standard V2 tier)
  • Supports multiple backend pools with different routing rules
Users → [App Gateway :80/:443]
         └── /api/*    → Backend Pool: API Servers
         └── /images/* → Backend Pool: Image Servers
         └── /*        → Backend Pool: Web Servers

Key Takeaways

  • Azure Load Balancer distributes traffic at Layer 4 (TCP/UDP) using hash-based distribution across backend pool members.
  • Azure Application Gateway manages traffic at Layer 7 (HTTP/HTTPS) with URL path-based routing and content-aware features.
  • Public load balancers distribute internet traffic; internal load balancers distribute traffic within a VNet.
  • Path-based routing enables directing requests to content-specific backend pools based on URL patterns.
  • The WAF tier on Application Gateway adds protection against common web vulnerabilities and malicious traffic filtering.

Exercises

Easy

Exercise 1: Load Balancer Health Check

  1. Navigate to an existing Load Balancer in the Azure Portal.
  2. Check the Backend pool health status.
  3. Review the health probe configuration: protocol, port, and interval.
  4. Identify: What happens to traffic if one backend VM goes down?
Show Hint
The health probe continuously checks backend health at the configured interval. When a VM fails the probe threshold, it is automatically removed from rotation. Traffic is distributed only to healthy VMs until the unhealthy VM recovers and passes probes again.

Exercise 2: Load Balancing Comparison

  1. Review Azure's "Load balancing — help me choose" tool in the portal.
  2. Compare: When would you use Load Balancer vs Application Gateway vs Traffic Manager vs Front Door?
  3. Create a table documenting: OSI layer, protocol support, global vs regional, and key features for each service.
  4. For your current lab setup, justify which load balancing solution fits best.
Show Hint
Load Balancer = Layer 4 regional (TCP/UDP hash-based distribution). Application Gateway = Layer 7 regional (HTTP path-based routing, SSL termination, WAF). Traffic Manager = DNS-based global (geographic/performance/priority routing). Front Door = Layer 7 global (CDN + WAF + global load balancing + SSL offload).
Hard

Exercise 3: Path-Based Routing Design

Design an Application Gateway configuration for an e-commerce site with these requirements:

  1. Main site at / serves from a pool of 3 web servers.
  2. /api/* routes to 2 API servers.
  3. /static/* routes to a storage-backed pool.
  4. /admin/* should only be accessible from the internal network (10.0.0.0/8).
  5. Draw or document the complete configuration: listeners, rules, backend pools, and health probes.
  6. Explain: How would you add SSL termination to this setup?
Show Hint
You would need 4 backend pools (web, api, storage, admin), a path-based routing rule with path maps for each URL pattern, and health probes tailored to each backend type. For the /admin/* restriction, use an NSG on the Application Gateway subnet or WAF custom rules. For SSL termination, upload a PFX certificate to the Application Gateway, configure an HTTPS listener on port 443, and optionally add an HTTP-to-HTTPS redirect rule on port 80.

Scenario

Your web app needs to route /api/* requests to backend API servers and /static/* to a CDN-origin pool. Which Azure service do you need?

AAzure Load Balancer — it can route by URL path
BAzure Traffic Manager — it does DNS-based routing
CAzure Application Gateway — it supports Layer 7 path-based routing
Why this is correctApplication Gateway operates at Layer 7 (HTTP) and supports path-based routing rules. Load Balancer is Layer 4 (TCP/UDP) — it can't inspect URLs. Traffic Manager is DNS-based and routes at the domain level, not the path level.
Common mistakesLoad Balancer (A) works at Layer 4 and has no visibility into HTTP URLs. Traffic Manager (B) resolves DNS to different endpoints — it can't split paths within the same domain.
Scenario

Your Load Balancer health probe is configured for TCP on port 80 with a 5-second interval. One backend VM's web server crashed but the VM itself is still running. Will the Load Balancer detect this?

ANo — TCP probes only check if the VM is running, not the web server
BYes — TCP probe checks if port 80 is accepting connections; a crashed web server won't accept connections
CNo — you need an HTTP probe to check web servers
Why this is correctA TCP health probe attempts a TCP connection to the specified port. If the web server process crashed, port 80 won't accept connections and the probe will fail, marking the VM as unhealthy. HTTP probes check for a specific status code (like 200) which adds application-level validation, but TCP probes DO detect crashed services.
Common mistakesTCP probes don't just ping the VM (A is wrong) — they attempt a real TCP handshake on the specified port. While HTTP probes offer deeper validation, TCP probes are sufficient for detecting a crashed service (C is wrong).
Scenario

You have a Standard SKU Load Balancer. Backend VMs are healthy but users report they can't access the site. The frontend public IP is correct. What should you check first?

ANSG rules — Standard LB requires explicit NSG rules to allow traffic
BThe VM operating system firewall
CWhether the VMs have public IPs
Why this is correctStandard SKU Load Balancers require an NSG rule to allow traffic. Unlike Basic SKU, Standard LB blocks all traffic by default until an NSG explicitly allows it. This is the most common gotcha when migrating from Basic to Standard.
Common mistakesWhile OS firewalls (B) can block traffic, the NSG is the first thing to check with Standard SKU. VMs behind a load balancer don't need public IPs (C) — the LB frontend IP handles inbound traffic.
! Troubleshooting Spot
Troubleshoot

Your Application Gateway is deployed and shows 2 healthy backends. But browsing to the frontend IP gives a 502 Bad Gateway error.

What's wrong?

▶ Hint 1
502 means the gateway received an invalid response from the backend. Check if the backend VMs are actually serving HTTP on the port specified in the backend settings.
▶ Hint 2
Check the backend settings — is the port configured to match what the web server is listening on? Also check if the health probe path returns a 200 status.
▶ Answer
The most common cause is a port mismatch — the backend HTTP settings specify a port (e.g., 80) but the web server is listening on a different port (e.g., 8080). Also verify the health probe path (e.g., "/") returns HTTP 200. If the probe path returns 404 or 500, the gateway marks backends as unhealthy even if the main app works.
◆ Architect's Corner
Design Decision

Your company's website gets traffic from users worldwide. You need: SSL termination, path-based routing for /api vs /web, WAF protection, and automatic failover between East US and West US regions. Which combination do you use?

Two Application Gateways (one per region) behind Azure Traffic Manager
DNS failover, regional App Gateways handle routing/WAF
Two Azure Load Balancers (one per region) behind Azure Front Door
Global L7 + regional L4
Azure Front Door with WAF, pointing to backend pools in both regions
Global L7 with built-in WAF, path routing, and health-based failover
Why this is the best choiceAzure Front Door is purpose-built for this scenario — global Layer 7 load balancing with built-in WAF, SSL termination, path-based routing, and automatic health-based failover between regions. Option A works but adds DNS-level latency for failover (TTL-dependent). Option B uses L4 Load Balancers regionally which can't do path-based routing. Front Door handles everything in one service.
