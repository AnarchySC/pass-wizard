What You'll Learn

Virtual machines are the workhorse of Azure compute. This module covers creating VMs, scaling them (up and out), managing disks, and using VM Scale Sets for automated scaling.

  • Deploy VMs across availability zones for high availability
  • Scale VMs vertically (resize) and horizontally (more instances)
  • Manage and change VM disks
  • Configure VM Scale Sets with autoscale rules
🖥

VM Fundamentals

When you create a VM, you choose four things:

  1. Image — The operating system (Windows Server, Ubuntu, etc.)
  2. Size — CPU, RAM, disk speed (B-series burstable, D-series general, E-series memory-optimized)
  3. Region — Where the VM physically runs
  4. Networking — VNet, subnet, public IP, NSG

A VM creates several sub-resources: a NIC (network interface), an OS disk, and optionally a public IP. These are separate resources that can be managed independently.

Note: VM sizes determine your SLA, performance, and cost. B-series VMs are the cheapest but have variable CPU performance (they 'burst'). For production workloads, use D-series or higher.
✅

Availability and SLA

Azure guarantees different SLA levels depending on your architecture:

ArchitectureSLAWhat it means
Single VM (Premium SSD)99.9%Up to 8.76 hours downtime/year
Availability Set99.95%Up to 4.38 hours downtime/year
Availability Zones99.99%Up to 52.6 minutes downtime/year

Availability Set — VMs spread across fault domains (separate physical racks) and update domains (separate maintenance groups) within ONE datacenter.

Availability Zones — VMs spread across physically separate datacenters within a region. Each zone has independent power, cooling, and networking.

Real-World: If your app needs high availability, deploy at least 2 VMs across Availability Zones with a Load Balancer. This gives you 99.99% SLA and protects against entire datacenter failures.
📈

Scaling

Two types of scaling:

Vertical scaling (scale up/down) — Change the VM size. More CPU, more RAM. Requires a restart. Good for predictable, steady growth.

Horizontal scaling (scale out/in) — Add or remove VM instances. No downtime for existing VMs. Good for variable workloads.

🛠

VM Scale Sets

A VM Scale Set (VMSS) is a group of identical VMs managed as a unit. Instead of creating VMs one by one, you define a template and Azure handles the rest.

Key features:

  • Autoscale — Automatically add/remove instances based on metrics (CPU, memory, queue length)
  • Rolling upgrades — Update VMs without downtime
  • Zone spanning — Spread instances across availability zones

Autoscale rules have two parts:

  • Scale-out rule: When CPU > 70% for 10 minutes → add 50% more instances
  • Scale-in rule: When CPU < 30% for 10 minutes → remove 50% of instances
  • Cooldown period: Wait 5 minutes after scaling before evaluating again (prevents thrashing)
Warning: Always configure BOTH scale-out AND scale-in rules. If you only configure scale-out, instances are added during load spikes but never removed — you'll pay for idle VMs forever.

Lab 08 — Manage Virtual Machines

Estimated time: 50 minutes

1

Deploy Zone-Resilient VMs

▼

Create two virtual machines deployed across availability zones for high availability.

  1. Navigate to Virtual Machines and click + Create → Azure virtual machine.
  2. Configure the first VM:
    SettingValue
    Resource Groupaz104-rg8 (create new)
    Nameaz104-vm1
    Availability ZoneZone 1
    ImageWindows Server 2025
    SizeStandard_D2s_v3
  3. On the Disks tab:
    SettingValue
    OS disk typePremium SSD
    Delete with VMChecked
  4. On the Networking tab, set Public inbound ports to None.
  5. On the Management tab, disable Boot diagnostics.
  6. Click Review + create, then Create.
  7. Repeat to create az104-vm2 with the same settings but in Availability Zone 2.
Note Deploying VMs across availability zones provides 99.99% uptime SLA. Each zone is a separate physical datacenter within the same Azure region.
2

Manage Compute and Storage Scaling

▼

Resize a VM and manage data disks including attaching, detaching, and upgrading.

  1. Navigate to az104-vm1 → Size blade.
  2. Resize the VM to Standard_D2ds_v4 and click Resize.
  3. Navigate to the Disks blade and click + Create and attach a new disk:
    SettingValue
    Disk namevm1-disk1
    Size32 GB
    Storage typeStandard HDD
  4. Click Apply to attach the disk.
  5. Return to the Disks blade and detach vm1-disk1 by clicking the X icon, then Apply.
  6. Search for the vm1-disk1 disk resource, open it, and navigate to Size + performance.
  7. Upgrade the storage type to Standard SSD and click Save.
  8. Return to az104-vm1 → Disks and re-attach the upgraded disk.
Tip You can change a disk's SKU (HDD, SSD, Premium SSD) only while it is unattached. Detach first, upgrade, then re-attach.
3

Create VM Scale Set

▼

Deploy a VM Scale Set with load balancing and a network security group.

  1. Navigate to Virtual machine scale sets and click + Create.
  2. Configure the basics:
    SettingValue
    Resource Groupaz104-rg8
    Namevmss1
    Availability ZonesZones 1, 2, 3
    Orchestration modeUniform
    ImageWindows Server 2025
    SizeStandard_D2s_v3
  3. On the Networking tab, configure the virtual network:
    SettingValue
    Virtual networkvmss-vnet (create new)
    Address range10.82.0.0/20
    Subnet namesubnet0
    Subnet range10.82.0.0/24
  4. Edit the network interface to create a new NSG named vmss1-nsg.
  5. Add an inbound rule to the NSG:
    SettingValue
    NameAllowHTTP
    Destination port80
    ProtocolTCP
    ActionAllow
    Priority1010
  6. Under Load balancing, select Azure load balancer and create a new one named vmss-lb.
  7. Click Review + create, then Create.
Note Uniform orchestration treats all instances identically — ideal for stateless workloads. Flexible orchestration allows mixing VM sizes and is better for heterogeneous workloads.
4

Scale VM Scale Sets

▼

Configure custom autoscale rules based on CPU metrics.

  1. Navigate to vmss1 → Scaling blade.
  2. Select Custom autoscale.
  3. Set the Scale mode to Scale based on a metric.
  4. Add a scale-out rule:
    SettingValue
    MetricPercentage CPU
    OperatorGreater than
    Threshold70%
    Duration10 minutes
    OperationIncrease percent by
    Value50%
  5. Add a scale-in rule:
    SettingValue
    MetricPercentage CPU
    OperatorLess than
    Threshold30%
    Duration10 minutes
    OperationDecrease percent by
    Value50%
  6. Set the instance limits:
    SettingValue
    Minimum2
    Maximum10
    Default2
  7. Click Save.
Tip Always configure both scale-out and scale-in rules. Without a scale-in rule, you may end up paying for idle instances that never get removed.
5

Create VM with PowerShell (Optional)

▼

Use Azure PowerShell to create a virtual machine from the command line.

  1. Open Azure Cloud Shell and select PowerShell.
  2. Run the following command to create a VM:
    PowerShell New-AzVm -ResourceGroupName 'az104-rg8' -Name 'myPSVM' -Location 'East US' -Image 'Win2019Datacenter' -Zone '1' -Size 'Standard_D2s_v3'
  3. When prompted, enter a username and password for the VM admin account.
  4. Once deployment completes, verify the VM appears in the az104-rg8 resource group.
Note PowerShell is ideal for scripted or repeatable deployments. The New-AzVm cmdlet creates a VM along with all required resources (NIC, disk, NSG) automatically.
6

Create VM with CLI (Optional)

▼

Use Azure CLI to create a Linux virtual machine.

  1. Open Azure Cloud Shell and select Bash.
  2. Run the following command to create a VM:
    CLI az vm create --name myCLIVM --resource-group az104-rg8 --image Ubuntu2204 --admin-username localadmin --generate-ssh-keys
  3. Wait for the deployment to complete. The output will include the public IP address.
  4. Verify the VM appears in the az104-rg8 resource group.
Tip The --generate-ssh-keys flag automatically creates an SSH key pair if one doesn't exist. The private key is stored in ~/.ssh/ in your Cloud Shell session.

Key Takeaways

  • Virtual machines provide on-demand, scalable computing resources in Azure.
  • Both vertical scaling (resizing VM size) and horizontal scaling (adding instances) are available.
  • VM Scale Sets enable load-balanced group management with autoscale rules.
  • Availability zones support 99.99% uptime SLA by distributing VMs across physical datacenters.

Connection Methods: RDP, SSH, Bastion

▼

Scaling Configuration: Manual and CPU-Based Autoscale

▼

  2. Switch to custom autoscale and walk through the rule builder.

Scenario

A single VM with a Premium SSD gives you 99.9% SLA. Your app needs 99.99%. What do you do?

AUpgrade to a larger VM size
BAdd a Premium SSD data disk
CDeploy two VMs across Availability Zones with a Load Balancer
Why this is correct99.99% SLA requires Availability Zones (VMs in separate physical datacenters within a region). VM size and disk type don't affect the SLA beyond the single-VM 99.9% for Premium SSD. Availability Sets give 99.95%. Only zones reach 99.99%.
Common mistakesVM size (A) affects performance, not availability SLA. Adding disks (B) affects storage capacity, not uptime guarantees. SLA is about redundancy architecture, not hardware specs.
Scenario

You resized a VM from D2s_v3 to D4s_v3 and it rebooted. Is this expected?

ANo — resizing should happen without downtime
BYes — resizing a VM requires a restart to allocate new hardware
COnly if you change VM families (e.g., D-series to E-series)
Why this is correctAll VM resize operations require a restart because Azure needs to move the VM to hardware that can support the new size. This applies even when staying in the same family. Plan for downtime or use scale sets for zero-downtime scaling.
Common mistakesThere is no live resize for Azure VMs (A is wrong). The restart happens regardless of whether you stay in the same family or switch families (C is wrong).
Scenario

Your VM Scale Set has autoscale: scale out at CPU > 70% with a 5-minute cooldown. After a CPU spike, 3 new instances were added. But CPU is now at 30% across all instances. What happens next?

ANothing immediately — the scale-in rule (if configured) will evaluate and eventually remove excess instances
BThe 3 new instances are immediately removed
CThe scale set keeps all instances forever until manually removed
Why this is correctScale-in is not automatic unless you've configured a scale-in rule. If you only configured scale-out, instances stay. If you have a scale-in rule (e.g., CPU < 30% for 10 min), it will evaluate after the cooldown and gradually remove instances. Instances are never immediately removed — the cooldown and evaluation period protect against thrashing.
Common mistakesInstances are never immediately removed (B is wrong) — cooldown periods prevent this. Instances won't stay forever IF a scale-in rule exists (C is wrong), but they will if you forgot to configure one.
! Troubleshooting Spot
Troubleshoot

You're trying to resize a VM to Standard_E4s_v3 but the size doesn't appear in the available sizes list. The VM is currently running in East US.

What's wrong?

▶ Hint 1
Not all VM sizes are available in every region. Check if E4s_v3 is supported in East US.
▶ Hint 2
Even if the size exists in the region, the specific hardware cluster your VM is on might not support it.
▶ Answer
When a VM is allocated to a hardware cluster, only sizes supported by that cluster appear. To access other sizes: deallocate the VM first (this releases it from the current cluster), then resize — you'll see the full list of sizes available in the region. If the size still doesn't appear, it's not available in that region at all.
◆ Architect's Corner
Design Decision

You're deploying a web application that gets variable traffic — 100 users during business hours, 10,000 during flash sales (unpredictable). What compute strategy do you use?

A single large VM that can handle peak load
Always on, pays for max capacity 24/7
A VM Scale Set with autoscale rules based on CPU and request count
Scales out for demand, scales in to save cost
A fixed pool of 20 VMs behind a load balancer
Handles moderate spikes, consistent cost
Why this is the best choiceVM Scale Sets with autoscale match your variable-traffic pattern perfectly. You pay for 2-3 instances during quiet periods and scale to 15+ during flash sales. Option A wastes money 90% of the time paying for capacity you don't need. Option C is better but still pays for 20 VMs during off-hours. Autoscale handles the unpredictability — configure scale-out on CPU > 70% AND custom metrics like HTTP request queue length for more responsive scaling.
