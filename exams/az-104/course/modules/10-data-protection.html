What You'll Learn

Data protection ensures you can recover from disasters, accidental deletions, and data corruption. This module covers Azure Backup for regular backups and Azure Site Recovery for disaster recovery replication.

  • Understand the difference between backup and disaster recovery
  • Create and configure Recovery Services vaults
  • Set up backup policies for VMs and file shares
  • Configure Azure Site Recovery for cross-region replication
🛡

Backup vs Disaster Recovery

These are related but different:

Backup — Take periodic snapshots of your data. If something goes wrong (accidental deletion, corruption, ransomware), restore from a backup. Recovery time: minutes to hours.

Disaster Recovery (DR) — Continuously replicate your entire environment to another region. If a region goes down, fail over to the replica. Recovery time: minutes.

AspectAzure BackupAzure Site Recovery
PurposeData recoveryBusiness continuity
FrequencyScheduled (daily, hourly)Continuous replication
RPOHours (since last backup)Minutes (near real-time)
RTOMinutes to hoursMinutes
ScopeIndividual resourcesEntire workloads
CostLowerHigher

RPO = Recovery Point Objective (how much data can you afford to lose). RTO = Recovery Time Objective (how fast must you recover).

🏦

Recovery Services Vaults

A Recovery Services vault is the storage container for all your backup data and replication metadata. It's the central management point for data protection.

Critical rules:

  • The vault must be in the same region as the resources it backs up
  • For disaster recovery, you create a second vault in the target region
  • Vaults support soft delete (14 days) — protects against accidental or malicious backup deletion
  • Vault redundancy (LRS, GRS) determines where backup data is stored
Always use GRS for production backup vaults. If your primary region goes down, your backups are safe in the secondary region. LRS is fine only for dev/test.
📅

Backup Policies

A backup policy defines WHEN and HOW LONG to keep backups:

  • Schedule — When to back up (daily at midnight, every 4 hours, etc.)
  • Retention — How long to keep each backup (daily: 30 days, weekly: 12 weeks, monthly: 12 months, yearly: 10 years)
  • Instant restore — Keep snapshots locally for fast recovery (1–5 days)

Azure Backup supports:

  • Azure VMs (full VM backup)
  • Azure Files (file share snapshots)
  • SQL Server in Azure VMs
  • Azure Blob Storage
🔄

Azure Site Recovery (ASR)

ASR continuously replicates VM disks to a secondary Azure region. If the primary region fails, you "fail over" — ASR creates running VMs from the replicated disks in the target region.

How it works:

  1. Enable replication on a VM → ASR installs the Mobility agent
  2. Disk writes are continuously replicated to cache storage in the source region
  3. Data is sent to managed disks in the target region
  4. Recovery points are created every few minutes
  5. During failover, Azure provisions VMs from the latest recovery point
Warning: ASR replicates at the DISK level. For databases, this can result in inconsistent state (partial transactions). Always use database-native replication (SQL Always On, Azure SQL geo-replication) for the data tier alongside ASR for compute.

Create Recovery Services Vault

▼

Create a Recovery Services vault to store backup data and manage backup policies.

  1. Search for and select Recovery Services vaults, then click + Create.
  2. Configure the vault:
    SettingValue
    Resource Groupaz104-rg-region1
    Vault nameaz104-rsv-region1
    RegionEast US
  3. Click Review + create, then Create.
  4. Once deployed, navigate to the vault and open the Properties blade.
  5. Under Backup Configuration, review the storage replication type (default: Geo-redundant).
  6. Under Security Settings, verify the soft delete retention period is set to 14 days.
Tip Geo-redundant storage replicates backup data to a paired Azure region, providing protection against regional outages. For cost savings in non-critical environments, you can switch to Locally-redundant storage — but only before the first backup is configured.

Enable VM Replication

▼

Configure Azure Site Recovery to replicate a VM to a secondary region for disaster recovery.

  1. Create a second Recovery Services vault in a different region:
    SettingValue
    Resource Groupaz104-rg-region2 (create new)
    Vault nameaz104-rsv-region2
    RegionWest US
  2. Navigate to az104-10-vm0 in the portal.
  3. Under Operations, select Disaster recovery.
  4. Configure the replication settings:
    SettingValue
    Target regionWest US
  5. Click Review + Start replication, then Start replication.
  6. Wait 10–15 minutes for the initial replication to complete.
  7. Return to the VM's Disaster recovery blade and verify the replication health shows Healthy.
Note Azure Site Recovery continuously replicates VM data to the target region. During a regional outage, you can fail over to the secondary region with minimal data loss. The target vault (az104-rsv-region2) must be in the same region as the failover destination.

Key Takeaways

  • Azure Backup provides simple, secure backup and recovery for Azure workloads.
  • Backup policies control the frequency of backups and retention of recovery points.
  • Azure Site Recovery replicates workloads to a secondary region for disaster recovery.
  • Recovery Services vaults centralize backup storage and management.
  • Soft delete protects against accidental deletion of backup data for 14 days.

Exercise 1: Backup Policy Review

  1. Navigate to a Recovery Services vault.
  2. Open Backup policies and review the default policy.

Show Hint
The default policy schedules daily backups at a specific time with 30-day retention. The maximum retention period for daily backups is 9999 days (approximately 27 years). Weekly, monthly, and yearly retention points can be configured separately for long-term compliance needs.

  3. Navigate to the backup item and select Restore VM.

Show Hint

Hard

Exercise 3: Disaster Recovery Architecture

Design a DR strategy for a 3-tier application (web tier, application tier, database tier).

  1. Identify the primary region (East US) and secondary region (West US).
  2. Configure Site Recovery replication for the web and application tier VMs.
  3. For the database tier, explain why Site Recovery alone is not sufficient.
  4. Document RPO (Recovery Point Objective) and RTO (Recovery Time Objective) targets for each tier.
  5. Create a failover runbook: What steps happen during failover? In what order?
  6. Explain the difference between planned failover and unplanned failover.
Show Hint
Databases need their own replication mechanism (e.g., SQL Always On Availability Groups, Cosmos DB multi-region writes) because Site Recovery handles compute replication, not data-layer consistency. Planned failover ensures zero data loss by synchronizing all pending changes before switching over. Unplanned failover may result in data loss up to the last recovery point (RPO). A typical runbook order: (1) fail over database, (2) fail over app tier, (3) fail over web tier, (4) update DNS/Traffic Manager.

? Quick Scenarios
Scenario

Your Recovery Services vault is in East US. Can you back up a VM in West US to this vault?

AYes — vaults can back up VMs in any region
BNo — the vault must be in the same region as the VM
CYes — but only with GRS redundancy enabled
Why this is correctRecovery Services vaults can only back up resources in the SAME region. To protect VMs in West US, you need a vault in West US. This is a hard requirement regardless of the vault's redundancy setting.
Common mistakesGRS replicates backup data to a paired region for redundancy, but it does not allow cross-region backup configuration. The vault and backed-up resource must always be co-located.
Scenario

You accidentally deleted a backup item from your Recovery Services vault. The data is gone immediately, right?

AYes — deletion is permanent and immediate
BNo — it moves to a recycle bin for 7 days
CNo — soft delete retains backup data for 14 additional days
Why this is correctSoft delete (enabled by default) keeps deleted backup data for 14 extra days. During this period, you can undelete the item and restore from it. This protects against accidental or malicious deletion. After 14 days, the data is permanently removed.
Common mistakesThere is no 7-day recycle bin. The retention period is specifically 14 days, and soft delete is enabled by default on all new vaults.
Scenario

Your company requires that production VMs can failover to another region within 15 minutes if the primary region goes down. What do you configure?

AAzure Site Recovery with continuous replication to a secondary region
BAzure Backup with hourly backup frequency
CVM snapshots taken every 15 minutes
Why this is correctSite Recovery continuously replicates VM disks to a secondary region. Failover takes minutes and creates running VMs in the target region.
Common mistakesBackup restores take much longer (must provision VMs and restore disks). Snapshots are point-in-time copies in the same region — they don't help if the region is down.
! Troubleshooting Spot
Troubleshoot

You're trying to delete a Recovery Services vault but get an error: "Vault cannot be deleted as there are existing resources within the vault."

What's wrong?

▶ Hint 1
You must stop protection and delete backup data for ALL backup items first. Check Backup Items for any remaining protected resources.
▶ Hint 2
Also check for registered infrastructure servers (Storage Accounts, SQL servers) under "Backup Infrastructure" and replication items under "Replicated Items".
▶ Answer
Vault deletion requires removing ALL dependencies in this order: 1) Stop backup and delete data for all backup items, 2) Unregister all backup infrastructure servers, 3) Delete all replicated items (Site Recovery), 4) Disable soft delete (otherwise deleted items still block vault deletion for 14 days), 5) Then delete the vault. Use the portal's "Delete" button — it will tell you exactly what's still blocking.
◆ Architect's Corner
Design Decision

Your company has a 3-tier app (web, app, database) in East US. Management wants DR with RPO of 1 hour and RTO of 30 minutes. How do you architect this?

Option A: Azure Backup only — daily backups of all VMs to a vault in West US
Simple setup, RPO = 24 hours, RTO = hours
Option B: Site Recovery for all 3 tiers replicating to West US
Continuous replication, fast failover, but database consistency risk
Option C: Site Recovery for web and app tiers + database-native replication (SQL Always On / geo-replication) for the data tier
ASR handles compute, database handles its own replication for consistency
Why this is the best choiceDatabases need their own replication mechanism because Site Recovery replicates at the disk level — it can't guarantee transactional consistency for an active database. SQL Always On or Azure SQL geo-replication provides transaction-consistent failover. ASR handles the web and app tier VMs. Option A has RPO of 24h (too slow). Option B risks database corruption during failover because disk-level replication can capture partial transactions.
