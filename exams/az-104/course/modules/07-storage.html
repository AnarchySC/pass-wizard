What You'll Learn

Azure Storage is one of the most used services in Azure. This module covers storage accounts, blob storage, file shares, security mechanisms, and how to optimize cost through lifecycle management.

  • Create and configure storage accounts with appropriate redundancy
  • Work with blob containers and understand access tiers
  • Secure storage with SAS tokens, firewalls, and service endpoints
  • Configure Azure Files for shared file storage
🗃

Storage Accounts

A storage account is a container for all your Azure Storage data. It provides a unique namespace accessible from anywhere via HTTP/HTTPS.

The key decision at creation: redundancy — how many copies of your data exist and where.

RedundancyCopiesGeographicRead AccessUse Case
LRS3 copies, 1 datacenterNoN/ADev/test, non-critical
ZRS3 copies, 3 zonesNoN/AHigh availability, same region
GRS6 copies, 2 regionsYesNo (failover only)Disaster recovery
RA-GRS6 copies, 2 regionsYesYes (read secondary)Mission-critical
Real-World: For production data you can't afford to lose, use at least GRS. For data you can recreate (build artifacts, caches), LRS is fine and much cheaper.
📦

Blob Storage and Access Tiers

Blob storage is for unstructured data — files, images, videos, backups, logs. Blobs live in containers (like folders).

Three access tiers optimize cost based on how often you access the data:

  • Hot — Frequently accessed. Highest storage cost, lowest access cost.
  • Cool — Infrequently accessed (30+ days). Lower storage cost, higher access cost.
  • Archive — Rarely accessed (180+ days). Cheapest storage, but retrieval takes hours and costs the most.

Lifecycle management automates tier transitions. Example: move to Cool after 30 days, Archive after 90, delete after 365.

Immutable storage (WORM) prevents data from being modified or deleted for a set period. Used for compliance — financial records, legal holds, healthcare data.

🔒

Securing Storage

Three ways to authenticate to storage:

  1. Access keys — Two 512-bit keys that grant FULL access to the entire storage account. Like a master key. Never share these.
  2. SAS tokens — Shared Access Signatures. Time-limited, permission-scoped, can be restricted to specific resources. The right choice for temporary access.
  3. Entra ID (RBAC) — Role-based access using Azure identities. Most secure, auditable, no keys to manage.

Beyond authentication, you can restrict WHERE traffic comes from:

  • Firewall rules — Allow only specific IP addresses
  • Service endpoints — Allow only traffic from specific VNets
  • Private endpoints — Give the storage account a private IP inside your VNet (most secure)
📁

Azure Files

Azure Files provides fully managed SMB file shares in the cloud. They work exactly like on-premises network drives — you can mount them from Windows, Linux, or macOS.

Use cases: shared configuration files, lift-and-shift of on-premises file servers, persistent storage for containers.

Key differences from Blob:

  • Files supports SMB protocol (mount as drive letter)
  • Blob is for object storage (access via REST API or SDK)
  • Files supports snapshots for point-in-time recovery

Create and Configure a Storage Account

▼

Create a storage account with geo-redundant storage, configure network access, and set up lifecycle management.

  1. Navigate to Storage accounts and click + Create.
  2. Configure the basics:
    SettingValue
    Resource Groupaz104-rg7 (create new)
    Storage account namestorageaccount[unique]
    RegionEast US
    PerformanceStandard
    RedundancyGeo-redundant storage (GRS)
  3. On the Data protection tab, check Enable read access to data in the event of regional unavailability (RA-GRS).
  4. Click Review + create, then Create.
  5. Once deployed, go to the storage account. Under Networking, set Public network access to Disabled. Click Save.
  6. Change Public network access to Enabled from selected virtual networks and IP addresses.
  7. Under Firewall, check Add your client IP address and click Save.
  8. Navigate to Lifecycle management under Data management.
  9. Add a rule:
    SettingValue
    Rule nameMoveToCool
    Rule scopeApply rule to all blobs
    Blob typeBlock blobs
    ConditionIf last modified more than 30 days ago
    ActionMove to cool storage
  10. Click Add to save the lifecycle rule.
Note Lifecycle management rules run once per day. Moving blobs to cool storage reduces costs for infrequently accessed data while keeping it immediately available.

Create and Configure Secure Blob Storage

▼

Create a blob container with private access, add an immutable storage policy, upload a blob, and test access with SAS tokens.

  1. In your storage account, navigate to Containers and click + Container.
  2. Create a container:
    SettingValue
    Namedata
    Public access levelPrivate (no anonymous access)
  3. Select the data container. Under Access policy, add an immutable storage policy:
    SettingValue
    Policy typeTime-based retention
    Retention period180 days
  4. Click Upload. In the Advanced section:
    SettingValue
    Upload to foldersecuritytest
    Blob typeBlock blob
    Access tierHot
  5. Select the uploaded blob and copy the URL.
  6. Test: Open an InPrivate/Incognito browser window and paste the URL. You should receive a ResourceNotFound or AuthenticationFailed error because the container is private.
  7. Go back to the blob and click Generate SAS:
    SettingValue
    PermissionsRead
    Start date/timeYesterday
    Expiry date/timeTomorrow (+2 days from start)
  8. Click Generate SAS token and URL. Copy the Blob SAS URL.
  9. Test: Paste the SAS URL in the InPrivate browser. The blob should download or display successfully.
Tip SAS tokens provide delegated, time-limited access to storage resources without sharing account keys. Always use the shortest practical expiry window.

Create and Configure Azure File Storage

▼

Create a file share, restrict access to a virtual network using service endpoints, and verify access control.

  1. In your storage account, navigate to File shares and click + File share.
  2. Create the file share:
    SettingValue
    Nameshare1
    TierTransaction optimized
  3. Open the Storage Browser and navigate to File shares > share1.
  4. Click Upload and upload a test file.
  5. Navigate to Virtual networks and create a new VNet:
    SettingValue
    Namevnet1
    Resource Groupaz104-rg7
    RegionEast US
    Address space10.0.0.0/16
    Subnet namedefault
    Subnet range10.0.0.0/24
  6. On the subnet, enable the Microsoft.Storage service endpoint.
  7. Return to the storage account Networking settings.
  8. Under Virtual networks, click + Add existing virtual network and select vnet1/default.
  9. Under Firewall, remove your client IP address. Click Save.
  10. Test: Open the Storage Browser and try to access share1. You should see a not authorized error, confirming that access is now restricted to the VNet only.
Warning After removing your client IP and restricting to VNet only, you will lose portal access to the storage account data. Re-add your IP to the firewall if you need to continue managing the account from the portal.

Key Takeaways

  • Storage accounts contain blobs, files, queues, and tables — all under a single account namespace.
  • Redundancy options include LRS (3 copies, one datacenter), ZRS (3 zones), GRS (6 copies, two regions), and RA-GRS (read access to secondary).
  • Immutable storage provides WORM (Write Once, Read Many) capabilities for regulatory compliance.
  • SAS tokens provide time-limited, scoped access to storage resources without exposing account keys.
  • Service endpoints restrict storage account access to traffic originating from specified virtual networks.

Exercises

Easy

Exercise 1: Storage Redundancy Comparison

  1. Create a storage account and review the Redundancy blade.
  2. Compare LRS, ZRS, GRS, RA-GRS, GZRS, and RA-GZRS options.
  3. For each, note: number of copies, geographic spread, and read access during an outage.
  4. Which would you choose for a mission-critical database backup? Why?
Show Hint
RA-GRS provides 6 copies across two regions with read access to the secondary region during outages. This gives the best durability and availability for critical backups. LRS has 3 copies in one datacenter, ZRS has 3 copies across zones, GRS has 6 copies across regions but no secondary read access.
Medium

Exercise 2: Blob Lifecycle Management

  1. Create a storage account with a blob container.
  2. Upload several test files.
  3. Configure a lifecycle management rule: move to Cool tier after 30 days, Archive after 90 days, delete after 365 days.
  4. Review the rule JSON in the portal.
  5. Explain: What happens to archived blobs? How do you access them?
Show Hint
Archived blobs are stored offline and cannot be read or modified directly. To access them, you must rehydrate them by changing the tier to Hot or Cool. Rehydration can take up to 15 hours with standard priority, or 1 hour with high priority (at higher cost).
Hard

Exercise 3: Secure Access Challenge

  1. Create a storage account and upload a sensitive file.
  2. Disable all public access.
  3. Create a VNet with a service endpoint for Microsoft.Storage.
  4. Configure the storage account firewall to allow only that VNet.
  5. Generate a SAS token with: read-only, 1-hour expiry, HTTPS only, restricted to a specific IP.
  6. Test access from allowed and disallowed sources.
  7. Explain the difference between SAS tokens, access keys, and Entra ID authentication.
Show Hint
Access keys give full, unrestricted control over the entire storage account — never share them. SAS tokens are delegated, time-limited, and scoped to specific permissions, services, and resources. Entra ID authentication uses RBAC roles (e.g., Storage Blob Data Reader) and is the most secure option because it does not require sharing secrets and supports conditional access policies.

Scenario

Your company stores compliance documents that must not be modified or deleted for 7 years. Which feature do you use?

ABlob soft delete with 2555-day retention
BImmutable blob storage with a time-based retention policy
CAzure Backup with a 7-year retention policy
Why this is correctImmutable storage enforces WORM (Write Once, Read Many) at the storage level. Data physically cannot be modified or deleted until the retention period expires. Soft delete only protects against accidental deletion (data CAN still be deleted permanently). Backup creates copies but doesn't prevent the original from being modified.
Common mistakesSoft delete (A) is a safety net, not a compliance lock — admins can still permanently delete data. Backup (C) protects against data loss but doesn't enforce immutability on the source data.
Scenario

A developer needs temporary read access to a specific blob for 2 hours. They shouldn't have ongoing access. What do you give them?

AThe storage account access key
BThe Reader RBAC role on the storage account
CA SAS token with read permission and 2-hour expiry
Why this is correctA SAS token is purpose-built for this — scoped to a specific resource, specific permissions, with a time limit. Access keys give full control over the entire account (way too much). RBAC Reader gives ongoing access you'd have to remember to revoke.
Common mistakesAccess keys (A) are the master keys to the entire storage account — never share them for temporary access. RBAC roles (B) don't expire automatically, creating a security risk if you forget to remove the assignment.
Scenario

You configured a lifecycle management rule to move blobs to Cool tier after 30 days. A blob was uploaded 45 days ago but it's still in Hot tier. Why?

AThe blob was modified or accessed within the last 30 days, resetting the timer
BLifecycle management rules only run monthly
CCool tier is full
Why this is correctLifecycle rules evaluate the "last modified" time. If the blob was accessed and the access tier was reset, or if the blob was modified, the 30-day clock restarts. Note: reads alone don't reset the timer unless you've also changed the blob. Also check that the rule actually targets the correct container/prefix.
Common mistakesLifecycle rules run daily, not monthly (B is wrong). Azure storage tiers don't have capacity limits (C is wrong) — Cool tier is a pricing tier, not a physical container.
! Troubleshooting Spot
Troubleshoot

You created a storage account, uploaded a blob, and shared the URL with a colleague. They get a 'ResourceNotFound' or 'PublicAccessNotPermitted' error when trying to access it.

What's wrong?

▶ Hint 1
Check the container's access level — if it's set to "Private", anonymous access is blocked.
▶ Hint 2
Check if the storage account has "Allow Blob public access" disabled at the account level (this overrides container settings).
▶ Answer
By default, new storage accounts disable public blob access at the account level. Even if you set the container to "Blob" or "Container" access level, the account-level setting blocks it. Either enable public access at the account level (not recommended for sensitive data) or generate a SAS token for your colleague.
◆ Architect's Corner
Design Decision

Your app generates 50TB of logs monthly. Logs are queried frequently for the first 7 days, occasionally for 30 days, then only for compliance audits for 1 year. How do you optimize cost?

Store everything in Hot tier for 1 year, then delete
Simple, fast access, highest cost
Lifecycle management: Hot for 7 days, Cool for 30 days, Archive after 30 days, delete after 365 days
Automatic tiering, cost-optimized
Store everything in Archive tier from day one
Cheapest storage, but retrieval takes hours
Why this is the best choiceLifecycle management automates the transition between tiers to match your access pattern. Hot tier for the first 7 days of active querying, Cool for occasional access, Archive for long-term compliance retention. This can reduce storage costs by 80%+ compared to keeping everything in Hot. Option A is simplest but most expensive. Option C is cheapest for storage but Archive retrieval takes up to 15 hours — unusable for the 7-day active querying window.
