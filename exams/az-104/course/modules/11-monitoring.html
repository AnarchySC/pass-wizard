What You'll Learn

Monitoring tells you what's happening in your Azure environment — before users complain. This module covers Azure Monitor, alerts, action groups, Log Analytics, and KQL queries.

  • Understand Azure Monitor's architecture (metrics, logs, alerts)
  • Create alert rules that detect problems automatically
  • Configure action groups for notifications (email, SMS, webhooks)
  • Query logs using KQL (Kusto Query Language)
📈

Azure Monitor Architecture

Azure Monitor collects two types of data from every Azure resource:

Metrics — Numerical values sampled at regular intervals (CPU %, disk IOPS, request count). Stored for 93 days. Fast to query, good for real-time dashboards and alerts.

Logs — Detailed records of events and operations (who logged in, which API was called, what errors occurred). Stored in a Log Analytics workspace. Rich querying with KQL.

Azure Resources → Azure Monitor
                   ├── Metrics → Alerts, Dashboards
                   └── Logs → Log Analytics → KQL Queries
                                  └── Alerts, Workbooks

Everything feeds into Azure Monitor automatically. You can also send custom metrics and logs from your applications.

🔔

Alerts and Action Groups

An alert rule has three parts:

  1. Scope — What to monitor (a VM, a subscription, a resource group)
  2. Condition — What triggers the alert (CPU > 80%, VM deleted, 5xx errors > 10)
  3. Action Group — Who gets notified and how

Action groups support multiple notification types:

  • Email — Good for records, bad for urgency
  • SMS — Immediate, hard to miss
  • Voice call — For critical, wake-up-at-3AM alerts
  • Webhook — Trigger external systems (PagerDuty, ServiceNow, Slack)
  • Azure Function / Logic App — Automated response

Alert severity levels:

  • Sev 0: Critical (entire system down)
  • Sev 1: Error (significant functionality impacted)
  • Sev 2: Warning (potential issue developing)
  • Sev 3: Informational (something noteworthy happened)
  • Sev 4: Verbose (detailed diagnostic info)
Use tiered notifications: Sev 0–1 → SMS + voice call to on-call. Sev 2 → email to team. Sev 3–4 → log only. Alert fatigue kills monitoring — if everything pages, nothing gets attention.
🔇

Alert Processing Rules

Alert processing rules let you suppress or modify notifications without disabling the alert rules themselves.

Use case: Planned maintenance every Sunday 2–4 AM. You don't want 50 alerts flooding your inbox during expected downtime.

An alert processing rule says: "For this scope, during this time window, suppress notifications." The alerts still fire and are recorded — you just don't get notified. When the maintenance window ends, notifications resume automatically.

🔍

Log Analytics and KQL

Log Analytics is where Azure stores all log data. You query it using KQL (Kusto Query Language) — a pipeline-based query language.

KQL basics — data flows left to right through pipes:

Heartbeat                              // Start with a table
| where TimeGenerated > ago(1h)        // Filter rows
| where Computer contains "web"        // More filtering
| summarize count() by Computer        // Aggregate
| sort by count_ desc                  // Sort
| render barchart                      // Visualize

Common tables:

  • Heartbeat — VM agent check-ins (is the VM alive?)
  • Perf — Performance counters (CPU, memory, disk)
  • AzureActivity — Control plane operations (who did what)
  • InsightsMetrics — VM Insights metrics
  • Syslog / Event — OS-level logs (Linux / Windows)
Tip: Start with pre-built queries in Log Analytics (Queries tab). They show you the available tables and common patterns. Modify them to learn KQL by example rather than from scratch.

Configure Alert Processing Rule

▼

Create a rule to suppress alert notifications during a planned maintenance window.

  1. Navigate to Monitor → Alerts → Alert processing rules.
  2. Click + Create and configure:
    SettingValue
    ActionSuppress notifications
    Schedule start10 PM tonight
    Schedule end7 AM tomorrow
    Rule namePlanned Maintenance
  3. Click Review + create, then Create.
Note Alert processing rules apply to already-fired alerts. They can suppress notifications or add action groups without modifying the original alert rules. This is ideal for planned maintenance windows.
6

Azure Monitor Log Queries

▼

Use KQL (Kusto Query Language) to query Azure Monitor logs.

  1. Navigate to Monitor → Logs and set the scope to your subscription.
  2. Run the pre-built Count heartbeats query from the query library.
  3. Switch to KQL mode for direct query editing.
  4. Run the following custom query to chart CPU utilization:
    KQL InsightsMetrics | where TimeGenerated > ago(1h) | where Name == "UtilizationPercentage" | summarize avg(Val) by bin(TimeGenerated, 5m), Computer | render timechart
  5. Review the resulting time chart showing CPU utilization per VM over the last hour.
Tip KQL processes queries top-to-bottom like a pipeline. Each line filters or transforms the data from the previous line. Use render as the final operator to visualize results.

Key Takeaways

  • Alerts enable early issue detection before user impact.
  • Action groups define notification recipients and automated responses.
  • Alert processing rules suppress notifications during planned maintenance.
  • KQL (Kusto Query Language) queries Azure Monitor logs for deep analysis.
  • Activity logs track all resource operations across your subscription.

Scenario

You created an alert rule for "VM Deleted" scoped to your subscription. A colleague deleted a VM in a different resource group than expected. Did the alert fire?

ANo — alerts only fire in the resource group where they're configured
BYes — the alert is scoped to the subscription, which includes all resource groups
CNo — delete alerts require specific VM targeting
Why this is correctAlert scope determines what gets monitored. Subscription scope covers ALL resource groups and resources within it. The alert fires for any VM deletion anywhere in the subscription.
Common mistakesConfusing alert scope with alert rule resource group. The resource group where the alert rule lives is just for management — the scope determines what triggers the alert.
Scenario

You set up an email action group for critical alerts. During a weekend outage, the on-call engineer didn't get the email for 20 minutes. What should you add?

AA second email address as backup
BA webhook to create a Jira ticket
CSMS and/or voice call notifications to the action group
Why this is correctEmail is not reliable for urgent alerts — it can be delayed, filtered, or missed. SMS and voice calls provide immediate, hard-to-miss notifications. Best practice: use email for record-keeping AND SMS/voice for urgency.
Common mistakesA second email has the same delay problem. Webhooks to Jira are for tracking, not immediate notification. For critical alerts, you need push-based channels like SMS or voice.
Scenario

Your team gets 50+ alert emails every night during scheduled maintenance. Most are routine and can be ignored. How do you fix this without disabling the alerts?

ACreate an alert processing rule that suppresses notifications during the maintenance window
BDelete the alert rules and recreate them after maintenance
CAdd a filter to the email action group to ignore certain alerts
Why this is correctAlert processing rules can suppress notifications on a schedule (e.g., Sunday 2-4 AM) without disabling the actual alert rules. The alerts still fire and are recorded — you just don't get notified.
Common mistakesDeleting rules loses monitoring coverage. Email filters are fragile and outside Azure's control. Alert processing rules are the purpose-built solution for this exact scenario.
! Troubleshooting Spot
Troubleshoot

You wrote a KQL query in Log Analytics but it returns no data: Heartbeat | where Computer == 'myVM' | where TimeGenerated > ago(1h). The VM is running and was recently enabled for monitoring.

What's wrong?

▶ Hint 1
VM Insights / Log Analytics agent takes time to start sending data. If you just enabled it, wait 10-15 minutes for initial data ingestion.
▶ Hint 2
Check if the Computer name matches exactly — KQL string comparisons are case-sensitive. Try Heartbeat | where Computer contains 'myVM' instead.
▶ Answer
Two common issues: 1) The agent was just installed and hasn't sent data yet — initial ingestion takes 5-15 minutes, and some data types (like InsightsMetrics) take even longer. 2) The Computer name doesn't match exactly — use Heartbeat | distinct Computer to see the actual names in your workspace. Computer names often include the FQDN (e.g., "myVM.internal.cloudapp.net" instead of "myVM").
◆ Architect's Corner
Design Decision

You're setting up monitoring for a production environment with 20 VMs, 5 App Services, and 3 SQL databases. What's your monitoring strategy?

Option A: Azure Monitor alerts on each individual resource with email notifications
Granular, 28+ individual alert rules, email-only
Option B: Centralized Log Analytics workspace + metric alerts with tiered action groups (email for warnings, SMS+email for critical)
Unified monitoring, severity-based notifications, dashboards
Option C: Third-party monitoring tool (Datadog/New Relic) replacing Azure Monitor entirely
Powerful, but additional cost and integration overhead
Why this is the best choiceA centralized Log Analytics workspace gives you one place to query across all resources. Metric alerts catch threshold breaches (CPU, DTU, response time). Tiered action groups ensure critical issues get immediate attention (SMS/voice) while warnings only email. Build Azure dashboards with KQL queries for real-time visibility. Option A creates alert sprawl with no unified view. Option C can complement Azure Monitor but shouldn't replace it — Azure Monitor provides native integration and is already included in your Azure costs.
