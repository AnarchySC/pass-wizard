Administer PaaS Compute Options

Deploy web apps with deployment slots, run serverless containers with ACI, and manage microservices with Azure Container Apps.

What You'll Learn

Platform as a Service (PaaS) lets you deploy applications without managing the underlying infrastructure. This module covers three Azure PaaS compute options: App Service, Container Instances, and Container Apps — when to use each and how they differ.

  • Understand PaaS vs IaaS and when to choose each
  • Deploy web apps with Azure App Service and deployment slots
  • Run containerized workloads with Azure Container Instances (ACI)
  • Deploy microservices with Azure Container Apps
☁

PaaS removes operational burden. You don't patch servers, configure load balancers, or manage OS updates. The trade-off: less control over the underlying infrastructure.

🌐

Azure App Service

App Service is Azure's managed platform for web apps, APIs, and mobile backends. It supports .NET, Java, Node.js, Python, PHP, and Ruby.

Key concepts:

  • App Service Plan — The compute that powers your app. Defines the VM size, region, and pricing tier. Multiple apps can share one plan.
  • Deployment Slots — Separate environments (staging, production) for your app. You deploy to staging, test it, then swap to production with zero downtime. If something breaks, swap back instantly.
  • Autoscale — Automatically add or remove instances based on rules or schedules.

Pricing tiers that matter:

  • Free (F1) — Testing only. No SLA, no custom domain, no slots.
  • Basic (B1) — Manual scaling. No slots.
  • Standard (S1) — Autoscale, 5 deployment slots, custom domains + SSL.
  • Premium (P1V3) — Enhanced performance, 20 slots, VNet integration.
Tip: Deployment slots are one of App Service's best features. Deploy to staging, test, swap. If the swap goes wrong, swap back. The old version is always one swap away.
📦

Azure Container Instances (ACI)

ACI is the simplest way to run a container in Azure. No VMs to manage, no orchestration to configure. You give Azure a container image, and it runs it.

Best for:

  • Short-lived tasks (batch jobs, data processing)
  • Simple applications that don't need scaling
  • CI/CD build agents
  • Quick testing and prototyping

NOT best for: Long-running services that need autoscaling, service discovery, or complex networking — use Container Apps or AKS instead.

ACI charges per-second. A container that runs for 10 minutes costs almost nothing. This makes it perfect for scheduled batch jobs — spin up, process, shut down.
🛠

Azure Container Apps

Container Apps is Azure's managed container platform built on Kubernetes — but you never touch Kubernetes directly. It handles scaling, load balancing, and service discovery for you.

Key features:

  • Revision-based deployment — Each deployment creates a revision. You can split traffic between revisions (canary deployments).
  • Scale to zero — Apps with no traffic scale down to zero instances (you pay nothing).
  • Dapr integration — Built-in microservices building blocks (service calls, pub/sub, state management).
  • KEDA-based scaling — Scale on HTTP requests, queue messages, cron schedules, and more.

Lab 09a — Implement Web Apps

Estimated time: 20 minutes

1

Create and Configure Azure Web App

▼

Create an App Service Web App with the PHP runtime on a Linux App Service Plan.

  1. Navigate to App Services and click + Create → Web App.
  2. Configure the Web App:
    SettingValue
    Resource Groupaz104-rg9 (create new)
    NameA globally unique name
    Runtime stackPHP 8.2
    Operating systemLinux
    Pricing planPremium V3 P1V3
  3. Click Review + create, then Create.
  4. Once deployed, navigate to the Web App and note the Default domain URL.
Note The Premium V3 tier provides enhanced performance, VNet integration, and deployment slots. Web App names must be globally unique as they form part of the URL.
2

Create Deployment Slot

▼

Add a staging deployment slot to enable safe pre-production testing.

  1. Navigate to your Web App → Deployment slots blade.
  2. Click + Add slot.
  3. Enter the name staging.
  4. For Clone settings from, select Do not clone settings.
  5. Click Add.
  6. Click on the staging slot to open it.
  7. Note the Default domain URL — verify it differs from the production slot URL (it will include -staging in the hostname).
Tip Deployment slots are live apps with their own hostnames. Cloning settings copies app settings and connection strings from an existing slot, which is useful when you want staging to mirror production configuration.
3

Configure Deployment Settings

▼

Deploy a sample PHP application to the staging slot using an external Git repository.

  1. In the staging slot, navigate to Deployment Center.
  2. Set Source to External Git.
  3. Configure the repository:
    SettingValue
    Repositoryhttps://github.com/Azure-Samples/php-docs-hello-world
    Branchmaster
  4. Click Save.
  5. Wait for the deployment to complete (check the Logs tab).
  6. Browse to the staging slot URL and verify you see “Hello World!”.
Note External Git deployment pulls code from a public repository. For private repos, use GitHub Actions, Azure DevOps, or local Git with deployment credentials.
4

Swap Deployment Slots

▼

Swap the staging and production slots to promote the new code to production.

  1. Navigate back to the main Web App (production slot).
  2. Go to the Deployment slots blade.
  3. Click Swap.
  4. Verify the Source is staging and Target is production.
  5. Click Start Swap and wait for completion.
  6. Browse to the production URL and verify it now shows “Hello World!”.
Tip Slot swaps are instant with no downtime. If the new version has issues, simply swap again to roll back. The swap operation exchanges the content and configuration between slots.
5

Configure and Test Autoscaling

▼

Enable automatic scaling and use Azure Load Testing to observe scaling behavior.

  1. Navigate to your Web App → Scale out (App Service plan).
  2. Select Automatic scaling.
  3. Set Maximum burst to 2.
  4. Click Save.
  5. Navigate to Azure Load Testing (search in the portal) and create a new load test resource.
  6. Create a quick test targeting your Web App URL.
  7. Monitor the following metrics during the test:
    MetricWhat to Watch
    Virtual usersNumber of simulated concurrent users
    Response timeAverage latency per request
    Requests/secThroughput under load
Note Automatic scaling in App Service differs from custom autoscale rules. It uses platform-managed scaling decisions based on HTTP traffic patterns rather than metric-based rules you define.

Test and Verify Container Instance

▼

Confirm the container is running and accessible via its FQDN.

  1. Navigate to the az104-c1 container instance.

Key Takeaways

  • App Services enable rapid web app deployment with built-in CI/CD, scaling, and SSL.
  • Deployment slots allow staging and testing before swapping to production with zero downtime.
  • Azure Container Instances (ACI) provide serverless containers ideal for short-lived or burst workloads.
  • Azure Container Apps provide managed Kubernetes-style orchestration without cluster management overhead.
  • Autoscaling is available at both the App Service and Container Apps level to handle variable traffic.

  2. Compare: deployment time, configuration options, and networking setup.
  3. Access both via their URLs and verify they work correctly.

Show Hint
ACI = simple, short-lived tasks and burst workloads. Container Apps = microservices and long-running apps without managing Kubernetes. AKS = full Kubernetes control for complex, multi-service workloads requiring custom orchestration.
Hard

Exercise 3: Blue-Green Deployment

Implement a blue-green deployment strategy using App Service deployment slots.

  1. Create a Web App with a production slot running version 1 of your app.
  2. Create a staging slot and deploy version 2 of your app.
  3. Test version 2 by browsing to the staging URL.
  4. Configure traffic routing: send 10% of production traffic to the staging slot.
  5. Monitor both slots, then perform a full swap when confident.
  6. Explain: How does this differ from a canary deployment?
Show Hint
Blue-green swaps 100% of traffic at once between two identical environments. Canary gradually increases the traffic percentage to the new version (e.g., 10% → 25% → 50% → 100%). App Service supports both via the Traffic % setting on the Deployment slots blade.

Scenario

You deployed a new version of your web app to the staging slot and it's working perfectly. You swap to production. Five minutes later, users report errors. What's the fastest way to recover?

ARedeploy the old version from Git
BSwap the slots again — the old version is still in the staging slot
CDelete the production slot and recreate it
Why this is correctSlot swaps are reversible. After swapping, the old production code is now in the staging slot. Simply swap again to roll back instantly. This is the whole point of deployment slots — zero-downtime deployments with instant rollback.
Common mistakesRedeploying from Git takes minutes; re-swapping takes seconds. Deleting a production slot is destructive and unnecessary when you can simply swap back.
Scenario

You need to run a batch processing job that takes 10 minutes, runs once daily, and processes files from Blob Storage. Which compute option is most cost-effective?

AAn App Service Web App on a Basic plan running 24/7
BA Virtual Machine Scale Set with autoscale
CAzure Container Instances triggered by a Logic App or timer
Why this is correctACI charges only while the container runs — 10 minutes per day. ACI is purpose-built for short-lived, event-driven workloads.
Common mistakesApp Service charges 24/7 even when idle. VMSS is massive overkill for a single batch job.
Scenario

Your Container App is running but you need to deploy a new version without downtime. What feature supports this?

AContainer Apps supports revision-based deployments with traffic splitting
BYou must stop the app, update the image, and restart
CContainer Apps doesn't support zero-downtime deployments
Why this is correctContainer Apps creates a new revision for each deployment. You can split traffic (e.g., 90/10) between the old and new revisions, validate, then shift 100% to the new revision. This is similar to App Service deployment slots but built into the container platform.
Common mistakesContainer Apps absolutely supports zero-downtime deployments. Stopping the app to update is unnecessary and causes downtime.
! Troubleshooting Spot
Troubleshoot

You deployed a web app from an External Git repo (GitHub). The deployment shows "Success" in the Deployment Center, but browsing to the URL still shows the default Azure placeholder page.

What's wrong?

▶ Hint 1
Check if the deployment actually pulled the latest code. Click on the deployment entry to see the commit hash and logs.
▶ Hint 2
The deployment might have succeeded but the app isn't configured for the correct startup. Check the runtime stack — is it set to match your app (e.g., PHP, Node, Python)?
▶ Answer
Most common cause: the repository branch is wrong. Check that the branch name matches exactly (e.g., "main" vs "master"). Also verify the app's startup command is correct for your runtime. For PHP apps, ensure the document root points to the correct folder. Check Application Logs under "Diagnose and solve problems" for specific errors.
◆ Architect's Corner
Design Decision

Your team is migrating 3 microservices to Azure. Service A is a REST API, Service B processes background queues, Service C is a simple static website. Each has different scaling needs. What do you use?

Option A: Deploy all three as App Service Web Apps in one App Service Plan
Shared plan, simple management, all scale together
Option B: Service A and B on Container Apps (separate scaling), Service C on Static Web Apps
Per-service scaling, right tool for each workload
Option C: Deploy all three on AKS (Kubernetes)
Full control, complex to manage
Why this is the best choiceContainer Apps gives Service A and B independent scaling without Kubernetes complexity — A scales on HTTP requests, B scales on queue length. Static Web Apps is free/cheap and purpose-built for Service C with built-in CDN. Option A forces all services to scale together (wasteful) and a static site doesn't need a full app runtime. Option C is powerful but overkill unless you have a dedicated platform team to manage Kubernetes.
