What You'll Learn

This module is about automating resource deployment. Instead of clicking through the portal every time, you'll use templates to define infrastructure as code — repeatable, consistent, and version-controlled.

  • Understand what ARM templates are and why they matter
  • Read and modify JSON template and parameter files
  • Deploy templates via the portal, PowerShell, and Azure CLI
  • Understand Bicep as a friendlier alternative to JSON
💻

Why Infrastructure as Code?

Clicking through the Azure portal works for learning, but it's terrible for production:

  • Not repeatable — Can you recreate your exact setup after someone deletes it?
  • Not auditable — Who changed what, and when?
  • Error-prone — Miss one checkbox and your VM is misconfigured
  • Slow — Deploying 20 resources by hand takes hours

Infrastructure as Code (IaC) means defining your infrastructure in files. You describe WHAT you want (a VM with these specs, a network with these subnets) and Azure creates it. The same file always produces the same result.

📄

ARM Templates

Azure Resource Manager (ARM) templates are JSON files that define Azure resources. Every ARM template has this structure:

JSON { "$schema": "...", "contentVersion": "1.0.0.0", "parameters": { }, "variables": { }, "resources": [ ], "outputs": { } }

The parameters file is separate — it contains the values for each deployment. Same template, different parameters = different environments (dev, staging, prod).

You can export a template from ANY existing Azure resource. This is a great way to learn — create something in the portal, export it, then study the JSON.
🚀

Deployment Methods

You can deploy ARM templates three ways:

MethodCommandBest for
PortalUpload template → DeployOne-off deployments, learning
PowerShellNew-AzResourceGroupDeploymentWindows admins, scripting
Azure CLIaz deployment group createCross-platform, Bash users

All three produce the exact same result. The portal is visual but manual. PowerShell and CLI can be scripted and automated.

Cloud Shell gives you both PowerShell and Bash directly in the browser — no local installation needed.

⚡

Bicep — ARM Made Readable

Bicep is Microsoft's answer to "ARM templates are too verbose." It's a domain-specific language that compiles to ARM JSON. Everything you can do in ARM, you can do in Bicep.

ARM JSON (verbose):

JSON {"type": "Microsoft.Compute/disks", "apiVersion": "2021-04-01", "name": "[parameters('diskName')]", "location": "[parameters('location')]", "properties": {"diskSizeGB": 32}}

Bicep (clean):

Bicep resource disk 'Microsoft.Compute/disks@2021-04-01' = { name: diskName location: location properties: { diskSizeGB: 32 } }

You can convert between them: az bicep decompile (JSON → Bicep) and az bicep build (Bicep → JSON).

Lab 03 - Manage Azure Resources by Using ARM Templates

Estimated time: 50 minutes

1

Create an ARM Template

▼
  1. Create a managed disk named az104-disk1 in resource group az104-rg3 (East US region).
  2. Configure the disk: Standard HDD, 32 GiB, no redundancy.
  3. Once the disk is deployed, go to the resource and select Export template.
  4. Download both template.json and parameters.json.
Note Exporting a template from a deployed resource captures its current configuration as reusable JSON, including all properties you set during creation.
2

Edit and Redeploy Template

▼
  1. Navigate to Deploy a custom template in the Azure portal.
  2. Select Build your own template in the editor and load your downloaded template.json.
  3. Change all disk name references from az104-disk1 to az104-disk2.
  4. Update the parameters file to match the new disk name.
  5. Deploy to resource group az104-rg3.
  6. Verify both az104-disk1 and az104-disk2 exist in the resource group.
3

Configure Cloud Shell and Deploy with PowerShell

▼
  1. Open Cloud Shell and select PowerShell. Mount a storage account if prompted.
  2. Upload your template.json and parameters.json files to Cloud Shell.
  3. Edit the template to change the disk name to az104-disk3.
  4. Deploy the template using PowerShell:
PowerShell New-AzResourceGroupDeployment ` -ResourceGroupName az104-rg3 ` -TemplateFile template.json ` -TemplateParameterFile parameters.json

Verify the deployment:

PowerShell Get-AzDisk | ft Name,ResourceGroupName,Location,DiskSizeGb,ProvisioningState
4

Deploy with CLI

▼
  1. Switch to Bash in Cloud Shell.
  2. Edit the template to change the disk name to az104-disk4.
  3. Deploy the template using Azure CLI:
Bash az deployment group create \ --resource-group az104-rg3 \ --template-file template.json \ --parameters parameters.json

Verify the deployment:

Bash az disk list --resource-group az104-rg3 --output table
5

Deploy Using Bicep

▼
  1. Upload the azuredeploydisk.bicep file to Cloud Shell.
  2. Edit the Bicep file with the following values:
Parameter Value
managedDiskName az104-disk5
diskSizeinGiB 32
sku StandardSSD_LRS

Deploy using Azure CLI:

Bash az deployment group create \ --resource-group az104-rg3 \ --template-file azuredeploydisk.bicep
Tip Bicep files do not require a separate parameters file. Parameters can be defined with defaults directly in the .bicep file.

Key Takeaways

  • ARM templates enable group-based resource deployment as a single operation.
  • Separate parameter files enhance template flexibility and reusability across environments.
  • ARM templates can be deployed via the Azure portal, PowerShell, or CLI.
  • Bicep provides concise, type-safe syntax for infrastructure-as-code with less boilerplate than JSON.

Easy

Exercise 1: Template Exploration

  1. Go to the Azure Quickstart Templates gallery.
  2. Find a template for deploying a simple storage account.
  3. Review the template.json — identify the resources, parameters, and variables sections.
  4. Note what values are parameterized vs hardcoded.
Hint (click to reveal)
Every ARM template has these sections: $schema, contentVersion, parameters, variables, resources, outputs.
Medium

Exercise 2: Cloud Shell Mastery

  1. Open Cloud Shell and switch between PowerShell and Bash.
  2. In PowerShell, list all resource groups:
    PowerShellGet-AzResourceGroup
  3. In Bash, list all resource groups:
    Bashaz group list --output table
  4. Create a resource group using CLI:
    Bashaz group create --name test-rg --location eastus
  5. Delete it using PowerShell:
    PowerShellRemove-AzResourceGroup -Name test-rg
Hint (click to reveal)
Cloud Shell persists files in an Azure File Share. Both shells can manage the same resources — choose based on your scripting preference.
Hard

Exercise 3: Build Your Own Template

Create an ARM template from scratch that deploys a virtual network.

  1. Define parameters for: vnetName, addressPrefix, subnetName, subnetPrefix.
  2. Define a Microsoft.Network/virtualNetworks resource.
  3. Deploy using Cloud Shell.
  4. Export the deployed resource's template and compare with yours.
  5. Convert your JSON template to Bicep syntax:
    Bashaz bicep decompile --file template.json
Hint (click to reveal)
Use az bicep decompile --file template.json to auto-convert JSON to Bicep. Compare the output to understand how Bicep simplifies the template syntax.

Scenario

You deployed a VM using an ARM template and need to deploy an identical one in another region. You have the template.json and parameters.json files. What's the fastest approach?

AManually create the VM in the portal matching every setting
BEdit the parameters file to change the region, then redeploy the template
CExport the existing VM as a new template
Why this is correctARM templates are reusable by design. Changing parameters (region, name) and redeploying is exactly what they're built for — fast, consistent, repeatable.
Common mistakesManually recreating (A) is slow and error-prone. Re-exporting (C) generates a new template that may include runtime-specific settings you don't need.
Scenario

Your team uses ARM templates (JSON) but finds them verbose and hard to read. A colleague suggests switching to Bicep. What's the relationship between Bicep and ARM?

ABicep is a completely different deployment system
BBicep templates can't do everything ARM templates can
CBicep compiles down to ARM JSON — it's syntactic sugar with full parity
Why this is correctBicep is a domain-specific language that transpiles to standard ARM JSON. Everything you can do in ARM, you can do in Bicep. The compiled output IS an ARM template.
Common mistakesBicep is NOT a separate system (A) — it's a layer on top of ARM. It has full parity (B) — you can even decompile JSON to Bicep with az bicep decompile.
Scenario

You're in Cloud Shell and need to deploy a template. You have both PowerShell and Bash available. Does it matter which you use?

ANo — both can deploy ARM templates, just with different syntax
BYes — only PowerShell can deploy ARM templates
CYes — Bash is required for Bicep templates
Why this is correctPowerShell uses New-AzResourceGroupDeployment, Bash uses az deployment group create. Both achieve the same result. Bicep works with both shells too.
Common mistakesBoth shells can deploy ARM and Bicep templates (B, C). The choice is about syntax preference, not capability.
! Troubleshooting Spot
Troubleshoot

You're deploying an ARM template and get the error: "The template parameter disk_name is not found." But you can see "disk_name" in your parameters file.

What's going wrong?

▶ Hint 1
ARM template parameter names are case-sensitive. Check if the template uses "disk_name" but your parameter file has "Disk_Name" or "diskName".
▶ Hint 2
Check if you edited the template to rename parameters but forgot to update the parameters file (or vice versa).
▶ Answer
Parameter names must match EXACTLY between template.json and parameters.json — including case and underscores. When you rename a parameter in the template, you must update it in the parameters file too. The error message tells you exactly which parameter it can't find.
◆ Architect's Corner
Design Decision

Your team deploys infrastructure weekly. Currently everyone clicks through the portal. You want to introduce Infrastructure as Code. What's your first step?

Option A
Rewrite everything in Terraform immediately. Full IaC solution, steep learning curve, external tool.
Option B
Export existing resources as ARM templates, then iterate toward Bicep. Start from what you have, learn gradually.
Option C
Write PowerShell scripts that call az commands. Familiar syntax, imperative approach.
Why this is the best choiceExporting existing resources gives you working templates immediately — no blank-page problem. You can then clean them up, parameterize them, and gradually convert to Bicep for readability. Terraform (A) is powerful but introduces a new ecosystem and state management. PowerShell scripts (C) are imperative — they describe HOW, not WHAT, making them harder to maintain and not idempotent by default.
